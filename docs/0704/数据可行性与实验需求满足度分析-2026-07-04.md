### 1.3 「满足」的两层定义与结论指针

沿用 0702/0703 的方法论，把"满足"拆成两层分别评判：

- **层 A（工程 / 流水线可复现性）**：数据能否驱动 摄取→落盘→预处理→弱标注→数据集→训练评估 全链路端到端跑通，隐私红线可验证，且本轮修复能被复测闭环。
- **层 B（科学 / 认证结论有效性）**：数据能否支撑论文级结论——真实 EER/AUROC/FAR/FRR、matched-impostor、无泄漏协议、按用户 bootstrap 统计。

**提前给出结论指针：层 A 满足，且为历次最干净（重置后 94 批、0 错误 0 隔离、契约 94/94、隐私红线全过、流水线一次跑通、修复前后复测闭环）；层 B 仍不满足——「单用户（0 冒充对）」「单日（`leave_day_out` 回退）」「金标单包（169 金标窗全在 `com.contextauth` 桶）」三条 P0 未破，且本轮每任务会话数多数=1，较 0703 是倒退。** 总判词维持"**工程可行 / 科学不可达**"；本轮的新增量不在数据规模，而在**修复闭环**与**信号缺失的定性归因**。

## 2. 结论速览（一页纸）

1. **总判定**：层 A（工程）**满足**；层 B（科学）**不满足**。一句话：**本轮数据把"流水线可复现 + 修复可复测"夯到了历次最好，但它是 1 名用户 1 天 7.8 分钟的冒烟数据——认证实验在数学上依旧无法定义。**
2. **规模**：94 batch = **35 金标（I0..I6 七类全覆盖）+ 59 第三方**；单次连续会话 7.8 min，金标墙钟合计 158s（2.6 min）；141,800 条三通道传感样本；102,600 个 UI 节点；572 条触控起止事件；实跑产出 **472 个 5s/1s 窗（169 金标）、204 维特征、`leakage_free=True`**。
3. **App 修复一（IMU 采样率，v1.1.0，已在落盘数据验证）**：0703 的 ~86Hz 主线程丢样已由 HandlerThread 修复根治——本批 accel/gyro **103.30Hz**、mag **100.00Hz**，逐批零离散、无 gap/burst/重复时间戳、`timestamp_elapsed_nanos` 单调违规=0、NaN=0。见 §5.3。
4. **App 修复二（I0 横屏 + 文案，v1.1.0，已验证）**：I0 `coarse_orientation` **landscape 227 / portrait 75（75% 横屏）**，`orient_landscape` 反相 bug 的旁证；I0 文案统一为 canonical **"Quiet watching and video"（5/5）**，无历史变体。
5. **App 修复三（v1.1.1，代码+单测已过，待上机）**：用户观察"详情页三种频率从 0 爬升到 100+"经定位为**纯 UI 显示层缺陷**（`SensorCollector.hz()` 用滑窗样本数÷硬编码 10.0，前 10s 分子线性增长 → 显示按真实速率×t/10 爬升），**落盘数据不受影响**（每任务首批即满速 103Hz）；已改为按样本时间戳跨度的瞬时速率（新增 `SensorRateMath`）。见 §4.1。
6. **App 修复四（v1.1.1，代码+单测已过，待上机）**：相邻批墙钟负间隔 91/93 对（最大 -6.05s）经定位为**混时钟基准**（传感样本墙钟加了 serverOffset、无障碍/快照/任务起点用未校正设备时间），**传感 elapsed 流完全连续、零样本重复**；新增 `ServerClock` 统一全部上传墙钟基准，字段名/schema 不变、服务端无需改动。见 §4.1、§5.6。
7. **Server 修复（research 层，本轮提交，已复测）**：修 `ui_surface_like`/`ui_bounds_occupancy` 量纲错配（除以像素屏面积 vs 真机 ÷24 网格 → 真机恒 0）、弱标注多处线索门控与类别归并、`split_manifest` 显式警告字段。**research/tests 72 + 摄取层 tests 56 全绿**。见 §4.2。
8. **复测闭环（金标 169 窗，基线→最终）**：弱标 top1 总一致率 **46.15% → 56.21%**；全集低置信占比 **23.9% → 17.4%**（113→82/472）；**I4 一致率 0 → 1.000**（`ScrollView`→webview 归并的直接结果）；I1 0.953 / I5 0.688 / I6 0.882 不变。见 §4.3。
9. **P0 之一未破——单用户**：真实设备唯一（`device_id` 1 个），冒充者=0 → 实跑三协议 **genuine/impostor 对数全为 0**，EER/AUROC 数学上不可计算，M0–M10 认证侧全不可达。
10. **P0 之二未破——单日**：`leave_day_out` 静默退化——实跑与 `leave_session_out` **逐窗相同**（`splits.py:235` 单日回退），本轮已新增 `warnings` 含 `leave_day_out_fell_back_to_leave_session_out` 显式可见。
11. **P0 之三未破——金标单包**：金标 100% 落在 `com.contextauth`（App 内置任务页）→ `leave_app_out` 把 **169 金标窗全部丢进 test**，train(xhs 243)/val(wechat 20) 金标为 0，跨 app 有监督训练不可行。
12. **I0/I2/I3 残留 0 一致率——定性为采集端信号缺失（非弱标注可调）**：金标批 `TYPE_VIEW_CLICKED=0`（Compose 任务页不发点击无障碍事件，全库 62 次点击全在第三方批）→ I2/I3 点击线索失效；I0 视频连发 `TYPE_WINDOW_CONTENT_CHANGED`（evt_rate≈11）破坏"低事件率"线索、轻运动使其被 I6 抢走；Compose `LazyColumn` 无 `RecyclerView` 类名 → `ui_list` 恒 0，I3 只剩与 I4 共享的 scroll。见 §4.3、§9.2。
13. **隐私红线全部通过**：102,600 节点的 `text/text_redacted/content_desc_redacted/window_title_redacted` 恒 null，0 个密码节点；端侧主动丢弃 28,309 文本节点 + 10,433 内容描述 + 548 可编辑文本；触控无坐标。drop-all-text 全项通过。
14. **下一步排序**：本轮已完成的采集端/流水线修复（IMU 速率、I0 横屏、Hz 显示、墙钟统一、surface 量纲、弱标注门控、可观测性）**均标 DONE**；未破的 P0 仍是 **多用户 + 多天**，新增 P0 是 **每任务多会话 + 分钟级时长**；新增 P1 是 **采集端点击/列表语义信号补强**（解 I2/I3）。见 §12。

## 3. 关键警示（务必先读）

1. **"跑通"≠"有效"，"修好显示"≠"有科学结论"**。本批数据可让 `run_preprocess → build_datasets` 全程无错退出、且修复前后可复测；但那是因为代码对数据不足**零硬性断言、全程静默退化**——冒充对为空、泄漏检查在空集上"空真"通过、指标返回 nan。本轮把"空真"改为**显式 `logger.warning` + `split_manifest` 警告字段**（`n_users`/`has_impostor_pairs`/`impostor_pool_check_vacuous`/`warnings`），但**警告可见不等于缺陷已解**——它只是让"数据不足"不再沉默。
2. **本轮弱标 56.21% 仍不足以直接充当 M7 弱监督**：`I0/I2/I3 一致率仍为 0`，且这三类的 0 是**采集端信号缺失**（点击事件不发、列表容器无原生类名、视频连发内容变更事件），**不是弱标注器可调**——在采集端补语义或接受域差声明之前，`M7` 的 KL 弱监督项质量不足。
3. **泄漏列已由代码强制剔除，但仍在原始数据里**：金标批 `estimated_context_category` 与 `task_category` **100% 恒等**（2970/2970），`coarse_orientation` 仅 I0 出现 landscape，`game_like_score` 仅 I6=0.8；本轮 `LEAKAGE_COLUMNS` 又**防御性补入** `media_like_score`/`list_like_score`/`form_like_score`。任何绕过白名单的自定义分析都会作弊。
4. **窗口不是独立样本**：5s 窗/1s 步重叠 80%，472 窗折算**独立 5s 金标窗仅约 31 个**（I0 4/I1 8/I2 2/I3 3/I4 3/I5 5/I6 3）；统计单位必须是"用户 / 用户×场景"（本批 = 1 名用户）。
5. **本轮每任务会话数多数=1，是相对 0703 的倒退**：`leave_session_out` 下金标随会话整包进入某折，跨会话结构薄弱；I5 唯一的 2 会话里还含 1 个 1.6s 碎片。正式采集必须回到"每任务 ≥2–3 session"。
6. **索引本轮干净**：`batches.jsonl` 94 行、`errors.jsonl` 0、`quarantine/` 空、`schema_validation` 94/94 = `ok`，无 append-only 残留（区别于 0703 的 334 行索引含 136 行墓碑）。**但路径陷阱仍在**：真实数据在 `deploy/data/paper/`，仓库根 `data/paper/` 是合成夹具，research 脚本默认读后者。
7. **v1.1.1 两项修复（`SensorRateMath`/`ServerClock`）尚未上机复采**：它们的正确性目前由**单元测试 + 数据侧反证**（落盘数据本就未受影响：首批即满速、elapsed 流零重叠）双重支撑，但"修复后的新数据长什么样"要等下一轮采集才能落地验证。本文对这两项一律标注"待上机验证"。

## 4. 本轮修复闭环（问题 → 根因 → 修复 → 复测）

### 4.1 App 侧（ContextAuthlab v1.1.0 → v1.1.1，versionCode 3；单测 68 个 / 22 类全绿；APK 已构建）

| # | 问题（现象） | 根因 | 修复 | 验证状态 |

- **P0-3 修复的可见性**：本轮 `split_manifest` 警告字段齐备（`n_users=1`、`has_impostor_pairs=false`、`impostor_pool_check_vacuous`、单日回退标志）——"数据不足"从静默变为显式，但**缺陷本身未解**（见 §3 警示 1）。

## 8. 核心论断一：工程 / 流水线可复现性（层 A）——**满足，且为历次最干净**

### 8.1 全链路在真实数据上一次跑通，且修复可复测

### 8.2 契约、隐私、修复闭环三重强化

契约由代码强制（94 金标批全过 `TASK_CATEGORIES` 契约）；泄漏列由 `LEAKAGE_COLUMNS` 白名单在特征/弱标两侧强制剔除，实跑 `leakage_free=True`；隐私红线全项通过（102,600 节点 0 文本泄漏、0 password）。本轮新增**修复闭环能力**：App 侧 v1.1.0 修复在落盘数据得到验证、v1.1.1 修复经单测 + 数据侧反证支撑；Server 侧修复经 research/tests 72 + 摄取层 tests 56 全绿 + 修复前后复测数字双证。

### 8.3 小结与可证伪点

**层 A 判定：满足。** 可证伪点：若在另一设备/另一天重复本流程而链路出现丢批、契约误杀或泄漏列残留，则层 A 判定需收回；若 v1.1.1 上机复采后 Hz 显示仍爬升或墙钟仍负间隔，则 §4.1 ①② 修复需重开。

## 9. 核心论断二：科学 / 认证结论有效性（层 B）——**不满足**

### 9.1 三条 P0 未破（结构性，不能靠同一用户多采弥补）

1. **单用户 → 冒充者=0**：认证是"本人 vs 他人"判别，负类整体缺失 → matched_impostor 对数=0、EER/AUROC/FAR/FRR 数学不可定义。0616、0702、0703、0704 四轮未变。
2. **单日 → 时间漂移不可评**：`leave_day_out` 逐窗回退（现有显式 warning）。
3. **金标单包 → 跨 app 有监督不可行**：169 金标窗全在 `com.contextauth` 桶，`leave_app_out` 把金标全丢进 test。

### 9.2 本轮的倒退与固有局限

- **每任务会话数倒退**：0703 的每场景 2–9 session 是当轮唯一被攻破的 P0；本轮多数任务仅 1 session（仅 I5=2 且含 1.6s 碎片），`leave_session_out` 再次接近 leave-one-class-out。**这是本轮定位为"冒烟采集"的直接代价，正式采集须回补。**
- **弱标 56.21% 仍不足以充当 M7 弱监督**：I0/I2/I3 一致率为 0，且如 §4.3 所述系**采集端信号缺失**（Compose 不发 CLICKED、`LazyColumn` 无 RecyclerView 类名、视频高频内容变更），非弱标注器可调——须在采集端补语义或接受域差声明。

### 9.3 窗口独立性与泄漏残面

5s/1s 重叠窗非独立样本，独立 5s 金标窗仅约 31 个；原始数据里 `estimated_context_category` 与 `task_category` 仍 100% 恒等（已被 `LEAKAGE_COLUMNS` 强制剔除，本轮又补入三列防御）。统计单位必须是"用户 / 用户×场景"（本批 n=1）。

### 9.4 小结与可证伪点

**层 B 判定：不满足。** 可证伪锚点：当出现 ≥2 名用户、≥2 天、每任务 ≥2–3 session 的数据，且在剔除泄漏列、enroll/query 跨 session 不相交、matched-impostor 协议下能算出非 nan 的 EER 及其按用户 bootstrap CI 时，本判定即被推翻——这正是 §12 P0 项的验收标准。

## 13. 结论

**回答标题问题：当前采集的数据，工程上满足、科学上不满足 `exp_prompt.md` 后续实验的需求。**

- **满足的部分**：数据结构与质量足以驱动 `exp_prompt` 第六~八节全部管线代码在真实数据上端到端运行（本文已实跑证实，且修复前后可复测）；隐私红线全项通过；7 类金标全非空；IMU 采样率回退已根治（86→103Hz）；本轮完成一条完整的**问题→根因→修复→复测**闭环（App v1.1.0 已验证 + v1.1.1 待上机；Server 本轮提交已复测）。RQ7（隐私/开销/部署）大部分可产出论文素材（§7.3 素材表）。
- **不满足的部分**：RQ1–RQ6 的全部落点（认证准确率类指标）在单用户、单日、金标单包数据上数学不可得——实跑三协议 genuine/impostor 对数均为 0、指标恒 nan；top-k 缺准确率轴；统计缺统计单位。本轮另有倒退：每任务会话数多为 1；弱标 56.21% 因 I0/I2/I3 采集端信号缺失仍不足以充当弱监督。
- **给评审者的一句话**：**本轮证明了"系统修得对、跑得干净、改得可复测"，但它是一次冒烟采集，论文的科学结论一条都还产不出来；在启动 20–40 人 × 3–5 天的多用户采集之前，先把「每任务多会话 + 分钟级时长」补上、把「采集端点击/列表语义」信号补强——否则大规模采集会带着 I0/I2/I3 同样的洞。**
