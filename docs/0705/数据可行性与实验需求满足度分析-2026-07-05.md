### 1.3 「满足」的两层定义与结论指针

## 2. 结论速览（一页纸）

1. **总判定**：层 A（工程）**满足**；层 B（科学）**不满足**。一句话：**本轮把"流水线可复现 + 隐私可验证 + 缺陷可修复"夯到了历次最好，但它仍是 1 名用户 1 天 37 分钟的冒烟数据——认证实验在数学上依旧无法定义。**
2. **规模**：120 batch = **61 金标（I0..I6 七类全覆盖）+ 59 第三方**；金标墙钟合计 158s；175,349 条三通道传感样本；116,415 个 UI 节点；582 条触控起止事件；修复后流水线产出 **469 个 5s/1s 窗（260 金标）、204 维特征、`leakage_free=True`**。
3. **v1.1.1 验证 A（IMU 采样率，已在落盘数据确认）**：accel/gyro **103.30Hz**、mag **100.00Hz**，逐批零离散、`timestamp_elapsed_nanos` 单调违规=0、重复=0、NaN=0、全零=0。0703 的 ~86Hz 主线程丢样已由 HandlerThread（v1.1.0）根治并在本批复现。
4. **v1.1.1 验证 B（ServerClock 墙钟统一，已确认，带保留）**：0704 旧数据的**混时钟病理签名（传感器墙钟加 serverOffset、事件未加 → ±6s 系统性重叠 + `batch_duration` 虚增）彻底消失**；残留的是**封套级**负间隔（117/119 相邻对为负，p50 −0.167s、最差 −2.587s），根因是 `ended_at` 封批延迟（p50 178.5ms）与第三方换批时新批 `started_at` 早于首样本（≤2.4s），**样本层零重叠零重复**——不是时钟基准问题。这作为采集健康度的已知残留项记录（见 §5.4、§9.2）。
5. **时钟根因反转（本轮重要澄清）**：观测到"120 批 `created_at` 一律比服务器 `ingested_at` 早约 226s"——经 App 代码级 + 数据级双查，**这不是 App 缺陷**：App 的 ServerClock 走 NTP 优先、设备钟准确，算出的 offset≈0 本就正确且全程被应用；真正错的是**部署宿主机时钟慢真实时间约 226–232s**（`timedatectl` 显示 `System clock synchronized: no`），只污染服务端 `ingested_at` 审计列，不影响批内数据自洽。**这是仓外运维项**（需在部署主机校时），并已由 App 端新增 `sync_source` 等 5 个诊断字段使此类问题今后可从落盘数据直接归因。
6. **🔴 隐私红线突破并已根治（本轮最重量级发现，SRV-1）**：既往数据的 `event_detail` 对文本类无障碍事件携带**逐次编辑的文本长度轨迹 + 光标位置**（如 `before_text_length` 23→24→28、`from_index` 光标位、121ms 级事件间隔），可重构击键动力学侧信道，**违反 `exp_prompt` §0.5/§3.1/§4.2「不采集长度序列 / 逐键时序」红线**。三管齐下修复：App 端删除 6 个长度/计数字段、不再调用 `getText()`、文本事件索引强制 -1 且时间戳量化整秒；Server 端 `ContextEvent` 增契约校验，旧 APK 若再上传即隔离；对在盘 120 批离线清洗（删 43,362 处禁用键、73 处文本时间戳整秒化，120/120 通过新契约校验）。
7. **隐私红线其余项全过**：116,415 UI 节点的 `text/text_redacted/content_desc_redacted/window_title_redacted` 恒 null、0 密码节点；端侧丢弃 31,761 文本节点 + 13,540 内容描述 + 585 可编辑文本；触控无坐标。drop-all-text 结构层全项通过。
8. **弱标注质量（修复后）**：金标窗 top1 一致率 **57.77%（基线）→ 63.85%（postfix，166/260）**；I1=1.000、I4=0.972、I6=0.923、I5=0.641；**I0=0.143、I2/I3=0** 系采集端 Compose 语义缺失固有局限（金标批 `TYPE_VIEW_CLICKED=0`、`LazyColumn` 无 `RecyclerView` 类名、视频高频 `WINDOW_CONTENT_CHANGED`），非弱标注器可调。
9. **包桶还原（SRV-2）**：基线预处理因会话级众数把包桶塌成 **2 个（错）**；修复后按窗口级众数 + 前台 app 变化切会话，还原为真实 **5 个**（com.contextauth / xhs / wechat / bili / miui.home），0 误标窗——`leave_app_out` / 包名特征 / M3 路由不再失真。
10. **P0 之一未破——单用户**：`device_id` 唯一，冒充者=0 → 三协议 genuine/impostor 对数全为 0，EER/AUROC 数学不可计算。
11. **P0 之二未破——单日**：`leave_day_out` 静默回退为 `leave_session_out`（本轮 warning `leave_day_out_fell_back_to_leave_session_out` 显式可见）。
12. **P0 之三未破——金标单包**：260 金标窗 100% 在 `com.contextauth`；`leave_app_out` 修复后正确识别其为金标承载 app 并整体留出 test，但 train/val 仍 0 金标——跨 app 有监督训练不可行。
13. **可观测性修复经真实数据验证**：三协议 `split_manifest` 均产出 `single_user_dataset` / `no_impostor_pairs_single_user_dataset`，`leave_day_out` 额外产出单日回退警告；新增 `user_coverage` / `impostor_windows_test_only` 字段——"数据不足"从静默变显式。
14. **下一步排序**：本轮已完成的采集端/流水线/隐私/论文级就绪性修复均标 DONE；未破的 P0 仍是 **多用户 + 多天 + 每任务多会话**；采集端 P1 是 **点击/列表语义信号补强**（解 I2/I3）。

## 3. 关键警示（务必先读）

1. **隐私红线在既往数据中曾被实际突破，现已根治**：这不是"潜在风险"而是**已发生**——0704/本轮既往落盘数据的 `event_detail` 确实携带了可重构击键节奏的文本长度序列。本轮已在 App 端删字段、Server 端加契约、在盘数据清洗三层堵死。任何引用既往（清洗前）数据快照的分析都须知悉该侧信道曾存在；引用本仓当前在盘数据是安全的（已清洗、120/120 过新契约）。**部署纪律**：App v1.1.2 与加了契约的 Server 镜像必须同批上线——Server 先行会把旧 APK（仍发禁用键）的上传全量隔离，这是预期的红线行为。
2. **"跑通"≠"有效"**：本批数据让 `run_preprocess → build_datasets` 全程无错退出、修复前后可复测；但那是因为代码对数据不足**零硬性断言、全程静默退化**——冒充对为空、指标返回 nan。本轮把"空真"改为显式 warning + manifest 字段，但**警告可见不等于缺陷已解**。
3. **弱标 63.85% 仍不足以直接充当 M7 弱监督**：`I0/I2/I3` 三类一致率低（0.143 / 0 / 0）且系采集端信号缺失，非弱标注器可调——须在采集端补语义或接受域差声明。
4. **泄漏列仍在原始数据里**：金标批 `estimated_context_category ≡ task_category` 100%，`coarse_orientation` 仅 I0 landscape，`game_like_score` 仅 I6=0.8；`LEAKAGE_COLUMNS` 已含 `media/list/form_like_score` 防御性剔除。任何绕过白名单的分析都会作弊。
5. **窗口不是独立样本**：5s 窗/1s 步重叠 80%，469 窗折算独立 5s 金标窗仅约 60 余个；统计单位必须是"用户 / 用户×场景"（本批 n=1）。
6. **本轮每任务会话数多数=1，仍是相对"每任务 ≥2–3 session"目标的缺口**：`leave_session_out` 下金标随会话整包进入某折（如 I3 金标仅存在于 test、I6 仅存在于 val）。正式采集必补。
7. **索引干净但路径陷阱仍在**：`index/batches.jsonl` 120 行、`errors.jsonl` 0、`quarantine/` 空。真实数据在 `deploy/data/paper/`，仓库根 `data/paper/` 是合成夹具，research 脚本默认读后者——跑真实数据必须 `--input deploy/data/paper`。
8. **部署宿主机时钟未同步**：慢约 226–232s，污染 `ingested_at`/`index` 时间戳。属仓外运维项，需在重部署前校时（见 §2.5、§9.2）。

## 8. 核心论断一：工程 / 流水线可复现性（层 A）——满足，且经本轮修复为历次最强

采集（7 任务金标 + 后台第三方）→ 摄取（契约 120/120 ok + 新增隐私红线拦截）→ 落盘（`devices/{date}` + `by_category/I0..I6` 软链一致）→ 预处理（120→469 窗）→ 数据集（3 协议 × 修复前后 2 套）全链无人工干预跑通；索引干净（120 行、0 残留、0 隔离）。本轮新增能力：隐私红线的结构契约拦截 + 在盘清洗 + 回归护栏三层；两仓库单测全绿（App 91/26、ingest 59、research 114）；修复前后复测数字双证。**层 A 判定：满足。**

可证伪点：若另一设备/另一天重复本流程出现丢批、契约误杀、泄漏列残留或隐私回潮，则层 A 判定需收回。

## 9. 核心论断二：科学 / 认证结论有效性（层 B）——不满足

### 9.1 三条 P0 未破（结构性，不能靠同一用户多采弥补）

1. **单用户 → 冒充者=0**：认证是"本人 vs 他人"判别，负类整体缺失 → matched_impostor 对数=0、EER/AUROC/FAR/FRR 数学不可定义。0616→0704→0705 五轮未变。
2. **单日 → 时间漂移不可评**：`leave_day_out` 逐窗回退（显式 warning）。
3. **金标单包 → 跨 app 有监督不可行**：260 金标窗全在 `com.contextauth`。

### 9.2 本轮的采集端固有局限与运维项

- **每任务会话数多为 1**：`leave_session_out` 接近 leave-one-class-out。
- **弱标 63.85% 因 I0/I2/I3 采集端信号缺失仍不足以充当弱监督**（Compose 不发 CLICKED、`LazyColumn` 无 RecyclerView 类名、视频高频内容变更）。
- **部署宿主机时钟未同步（运维项）**：慢约 226–232s，污染 `ingested_at`。需在部署主机校时（`sudo` 权限，仓外）。
- **墙钟封套级负间隔（采集健康度残留）**：不影响窗边界/特征（研究层用单调 elapsed），建议 App 端 `ended_at` 改用末样本墙钟。

### 9.3 小结与可证伪点

**层 B 判定：不满足。** 可证伪锚点：当出现 ≥2 名用户、≥2 天、每任务 ≥2–3 session 的数据，且在剔除泄漏列、enroll/query 跨 session 不相交、matched-impostor 协议下能算出非 nan 的 EER 及其按用户 bootstrap CI 时，本判定即被推翻——这正是 §11 P0 项的验收标准，且本轮训练面修复已使该口径就绪。

## 12. 结论

**回答标题问题：当前采集的数据，工程上满足、科学上不满足 `exp_prompt.md` 后续实验的需求。**

- **满足的部分**：数据结构与质量足以驱动全部管线代码在真实数据上端到端运行（本文已实跑证实，修复前后可复测）；隐私红线经本轮根治后全项通过；7 类金标全非空；IMU 103.3Hz 零丢样；本轮完成一条跨两仓库三战线的**问题→根因→修复→复测**闭环并使论文级流水线口径就绪。RQ7 大部分可产出论文素材。
- **不满足的部分**：RQ1–RQ6 的全部落点（认证准确率类指标）在单用户、单日、金标单包数据上数学不可得——三协议 genuine/impostor 对数均 0、指标恒 nan；top-k 缺准确率轴；统计缺统计单位。每任务会话数多为 1；弱标 63.85% 因 I0/I2/I3 采集端信号缺失仍不足以充当弱监督。
- **给评审者的一句话**：**本轮把系统修得对、跑得干净、改得可复测、隐私堵得严——但它仍是一次冒烟采集，论文的科学结论一条都还产不出来；在启动 20–40 人 × 3–5 天的多用户采集之前，先把「每任务多会话 + 分钟级时长」补上、把「采集端点击/列表语义」信号补强、并给部署主机校时——否则大规模采集会带着 I0/I2/I3 同样的洞。**
