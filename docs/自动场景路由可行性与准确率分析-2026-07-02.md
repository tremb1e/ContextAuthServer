## 2. 结论速览（先读三问）

1. **Q1 端侧数据够不够驱动自动路由？** **信号类型层面——够。** 路由器（弱标注/`weak_label`）读取的全部特征键 `LABEL_FEATURE_KEYS`（`interaction_states.py:46-82`）——事件计数、UI 节点结构标志、IMU 运动/姿态、触控频率——**都能从当前 AccessibilityEvent + UI 快照 + 三通道 IMU + 触控起止时间戳派生**，实跑已成功产出 204 维特征并逐窗打出 7 类概率。**缺的不是"信号种类"，而是**：①金标签只采在 App 自身任务页 UI（`com.contextauth`），UI 结构无法区分场景；②部分特征抽取质量差（`ui_surface_like` 恒 0、IMU 派生横屏 `orient_landscape` 反相）；③触控只有计数没有坐标；④学习式路由还需多用户。
2. **Q2 服务端代码能否实现自动场景路由？** **能，且分两条路径：**
   - **固定规则路由（可即刻运行、零训练）**：`weak_label`（`interaction_states.py:352`）→ `FixedRuleRouter = log(weak_probs)`（`routing.py:62-95`）→ `MoEAuthenticator._topk_gate` 取 top-k 并归一化门控（`moe.py:129-149`）→ 门控加权融合 7 专家嵌入。**这条链路今天就能把每个窗口自动路由到 top-k 专家。**
   - **学习式路由（已建但未训练）**：`LearnedRouter`（`routing.py:30-59`）需多用户数据训练，当前不可用。
   - **重要边界**：以上都在 `research/` **离线批处理管线**里（读 parquet）；**摄取服务 `app/main.py` 只有 6 个端点（health/ready/config/rules/ingest/metrics），没有任何在线路由/推理/认证端点**——"服务端自动路由"目前是离线脚本能力，不是实时 API。
3. **Q3 自动路由准确率多少、怎么算的？** **实测 top-1 = 23.31%**（top-2 = 64.66%、top-3 = 70.68%）。算法：`run_preprocess.py:255` 对每个 builtin 金标窗计算 `mean(weak_label_top1 == task_category)`——即固定规则路由的场景分类命中率（金标 = 受控任务 `task_category`）。**学习式路由没有"准确率"指标**（只在训练后产出 `expert_scene_matrix` 门控矩阵，`evaluator.py:157-170`，且需多用户）。**该 23.3% 无循环论证**（弱标注只读非泄漏特征，允许列表与 `LEAKAGE_COLUMNS` 断言不相交，`interaction_states.py:85`），但**金标本身是"受控任务意图"、且采于 App 任务页**（见 §6）。

> **提前给出结论指针**：**自动路由"能跑通"（Q1 信号够、Q2 代码有），但"跑不准"（Q3 top-1 仅 23.3%）**；准确率瓶颈是可定位、可修复的采集协议 + 特征质量 + 标注校准问题，而非缺少某类端侧信号。

## 3. 术语澄清：两种"路由器"、两种"置信度分数"、离线 vs 在线

需求原话「自动路由到 7 场景模型取 top-k 认证得出置信度分数」内含三处易混淆点，先厘清：

| 概念 | 含义 | 代码位置 | 当前可用性 |
|---|---|---|---|
| 固定规则路由器 | 手工打分函数把窗口特征 → 7 类概率（不训练） | `interaction_states.py:weak_label`、`routing.py:FixedRuleRouter` | **可即刻运行** |
| 学习式路由器 | 小 MLP 端到端学 `p(scene\|x)` | `routing.py:LearnedRouter`、由 MoE 训练 | 已建、**未训练**（需多用户） |
| top-k 门控 | 取路由概率最高的 k 个专家、权重归一后融合 | `moe.py:_topk_gate:129-149` | **可运行**（k∈{1..7}） |
| **路由置信度** | 路由决策的确信度 `clip(top1_prob − top4_prob,0,1)` | `interaction_states.py:389` | 可算（实测中位 0.304） |
| **认证置信度** | 查询嵌入与注册用户原型的 cosine 相似度（判定是否本人） | `evaluator.py:_cosine:120`、prototype 验证 | **需已训练编码器 + 多用户注册/冒充**，单用户下不可得 |

> **批判**：需求把"路由置信度"与"认证置信度"混为一谈。路由只决定**用哪几个专家算嵌入**；"是不是本人"的置信度是**嵌入与原型的 cosine**，来自认证评估阶段（`evaluator.py`），依赖训练好的模型与多用户注册库。本文 Q3 的"路由准确率"针对**路由器选对场景**这一环，不等于认证准确率（后者在单用户下无从定义，见 `docs/数据可行性与实验需求满足度分析-2026-07-02.md`）。

## 4. Q1 详解：端侧数据是否够驱动自动路由

### 4.1 路由器所需的每一路输入都能从当前数据派生（实测齐备）

弱标注（=固定规则路由）只允许读 `LABEL_FEATURE_KEYS`（`interaction_states.py:46-82`）。逐族核对其来源：

| 特征族 | 具体键 | 端侧来源 | 是否齐备 |
|---|---|---|---|
| 事件 | `evt_{click,longclick,scroll,textchanged,focus,windowstate,windowcontent}_count`、`evt_rate` | AccessibilityEvent 计数（`TYPE_VIEW_CLICKED/SCROLLED/TEXT_CHANGED/FOCUSED/WINDOW_*`） | **齐备**（实测 CLICK 15 367 / SCROLL 1 361 / EDIT 212 …） |
| UI 结构 | `ui_{node_count_mean,editable_count,scrollable_count,focusable_count,checked_count,selected_count,surface_like,webview,list,scroll_indicator,form_like_control_count,stable_ms}`、`ui_treediff_*` | UI 节点快照布尔/类名/bounds_grid + 快照间 tree diff | **齐备**（54 365 节点、29 类名、含 EditText/CheckBox/SeekBar/VideoView/WebView） |
| 运动/姿态 | `motion_energy_{low,mid,high}`、`gyro_burst_count`、`acc_mag_std`、`gyro_mag_mean`、`mag_mag_std`、`orient_landscape` | 三通道 IMU 原始逐样本 → 服务端特征层计算 | **齐备**（三通道 ~100 Hz 对等） |
| 触控 | `touch_rate` = 触控数/窗时长（`run_preprocess.py:66-68`） | `TOUCH_INTERACTION_START/END` 时间戳计数 | **齐备（仅计数）**（607 条） |

结论：**从"信号类型"看，AccessibilityEvent + UI 快照 + 三通道 IMU + 触控时间戳，足以计算路由器的全部输入**；drop-all-text（无文本）对路由**无影响**——路由器本就不读任何文本。

### 4.2 真正缺的四样东西（不是缺信号种类，而是缺"可区分场景的标注 + 特征质量"）

1. **【最关键】金标签采在 App 任务页 UI，不是真实场景 UI。** 实测 133 个金标窗**全部 `package_bucket = com.contextauth`**（App 自身任务页）；真实场景 UI（bilibili 等）只出现在**无金标签**的 183 个第三方窗里（`tv.danmaku.bili` 159 + `com.contextauth` 24）。于是 7 个受控任务在 UI 结构上**长得几乎一样**（都是任务页），UI 类打分函数无从区分场景（详见 §6）。
2. **部分 UI/姿态特征抽取质量差。** 金标窗 `ui_surface_like` **恒为 0.0（含 C5 视频、C6 画布）**——媒体/画布场景赖以区分的"大 surface"信号在数据上消失；`ui_node_count_mean` 恒 24–52（非 C6 期望的 <8）；IMU 派生 `orient_landscape` 在竖屏任务上恒 1.0、在真横屏的 C5 上反而只有 0.23（**反相**）。这些是**特征抽取/采集标定问题**，不是字段缺失。
3. **触控只有计数、无坐标。** 支撑 `touch_rate`（C6 高触控密度）够用，但做不了更细的触控节奏/轨迹路由证据。
4. **学习式路由需多用户。** 固定规则路由不需训练即可跑；但要 `LearnedRouter` 学出比手工规则更好的 `p(scene|x)`，需多用户、且理想情况下在真实场景 UI 上带标签的数据。

## 5. Q2 详解：服务端代码能否实现自动场景路由

**结论：能——固定规则路由链路今天即可端到端运行；学习式路由已实现但未训练；且当前只存在于离线管线、无在线端点。**

### 5.1 已实现的自动路由链路（逐段代码）

1. **打分 → 7 类概率**：`weak_label`（`interaction_states.py:352-405`）用 7 组加/减分 LF（`_score_c0…_score_c6`）→ 温度 softmax → `probs[7]`、`top1`、`topk`、`confidence`。
2. **概率 → 路由 logits**：`FixedRuleRouter.forward` 返回 `log(weak_probs).detach()`（`routing.py:78-95`），无需训练。
3. **top-k 稀疏门控**：`MoEAuthenticator._topk_gate`（`moe.py:129-149`）对路由概率取 `torch.topk(k)`、把留下的权重归一化到和为 1、其余置 0 → `gate_weights[7]`、`topk_indices[k]`。
4. **门控融合 7 专家**：`forward`（`moe.py:151-188`）`einsum("be,bed->bd")` 把 7 个专家嵌入按门控加权求和 → 融合嵌入 → 分类头/认证嵌入。
5. **k 可配 1..7**：`top_k` 校验在 `[1, 7]`（`moe.py:97-99`），即需求的"取 top-k 个模型"。

> **说明**：这条链路的**路由部分**（1→3 步）**不依赖训练、不依赖多用户**——把 App 上传数据经 `run_preprocess` 转成窗口特征后，即可对每个窗口自动路由到 top-k 专家。缺的是"专家/编码器"的**认证能力训练**（4→5 步的嵌入要有判别力需多用户，见 §3 认证置信度）。

### 5.2 两条路径与一条边界

- **固定规则路由**：可运行（M4 top-1 / M5 top-k baseline 即用它，`routing.py:8-9`）。
- **学习式路由**：`LearnedRouter` 已实现，但**需训练**；单用户数据训不出可泛化的路由（且认证损失退化，见可行性文档 §8）。
- **边界（务必知悉）**：`app/main.py` 仅 `@app.get(/health,/ready,/api/v1/config,/api/v1/rules,/metrics)` + `@app.post(/api/v1/ingest)`——**没有任何 `/route`、`/infer`、`/authenticate` 端点**。所谓"服务端自动路由"当前是 `research/` **离线批处理**（脚本读 `data/processed/*.parquet`），**尚未做成实时在线服务**。若要"上传即路由出分"，需另建在线推理端点（把 `run_preprocess` 的特征化 + `weak_label`/MoE 前向包成服务）。

## 6. Q3 详解：路由准确率与其计算方法

### 6.1 准确率如何计算（唯一 gold-referenced 路由指标）

`run_preprocess.py:252-255`：对所有金标窗（`task_category` 非空，仅 BUILTIN），
`gold_agreement = mean(weak_label_top1 == task_category)`。
即**固定规则路由 top-1 命中受控任务场景的比例**。金标 `task_category` 从窗口 IMU 行读出（`run_preprocess.py:114-120`）。**学习式路由无对应准确率**——训练后只有 `expert_scene_matrix`（按弱标签分组的平均门控权重，`evaluator.py:156-170`）、`expert_utilization`、路由熵，且需多用户模型。

### 6.2 实测准确率（对真实数据 133 金标窗复算）

| 指标 | 值 |
|---|---|
| top-1 路由准确率 | **0.2331（23.31%）**（与 `preprocess_report` 一致） |
| top-2 覆盖率（金标 ∈ top-2） | 0.6466 |
| top-3 覆盖率（金标 ∈ top-3） | 0.7068 |
| 金标窗路由置信度中位 / 低置信占比 | 0.304 / 18% |

> **含义**：因 MoE 实际是"取 top-k 专家"，**top-k 覆盖率比 top-1 更贴近该架构的有效性**——正确专家在 top-3 中的概率 70.7%，但仍被 C0/C3/C5 拖低。

### 6.3 逐场景准确率（实测）

| C1 文本输入 | KEYBOARD | 29 | **0.72** | **1.00** | C2×8 |
| C2 连续滑动 | SCROLLING | 5 | 0.60 | 0.80 | C0×2 |

### 6.4 为什么这么低——五个可定位根因（代码 + 数据双证）

1. **C2 标注函数过度触发（吸收池）。** C2 打分 `scroll_count>0 → +1.5`、`scrollable_count>0 → +0.9`、`list/webview/scroll_indicator → +0.8`（`interaction_states.py:184-192`）。几乎所有安卓 UI 都含可滚动容器，故 C2 分数在多数窗里胜出——全量 316 窗弱标签分布 C2 高达 209、C4 仅 4。**这是标注函数校准问题。**
2. **【核心】金标数据的 UI 不是场景 UI。** 133 金标窗全采于 App 任务页（`com.contextauth`），7 个任务的 UI 结构几乎相同。金标 `task_category` 反映的是"用户被要求做什么"（意图），而路由器看的是"屏幕结构像什么"（结构）——二者在任务页上系统性错位。真实场景 UI 只在**无标签**的第三方数据里。
3. **媒体/画布的关键 UI 线索在数据上消失。** 金标窗 `ui_surface_like` 恒 0.0（含 C5 视频任务），而 C5/C6 打分强依赖 `ui_surface_like>0.5`（`interaction_states.py:256,293`）→ 该正分从不触发；`ui_node_count_mean` 恒 24–52，C6 的 `node_count<8 → +0.5`（`:296`）也从不触发。
4. **IMU 派生横屏 `orient_landscape` 反相。** 实测竖屏任务（C0–C4、C6）`orient_landscape` 恒 1.0，真横屏的 C5 反而 0.23。C5 的 `orient_landscape>0.5 → +0.9`（`:274`）于是**在错误场景上加分、在正确场景上不加分**，直接压低 C5。**这是姿态特征标定 bug。**
5. **运动阈值未达。** C6 转腕运动 `motion_energy_high` 均值仅 0.09 < 阈值 0.3（`:299`），故 C6 高运动正分多半不触发——C6 top-1 塌向 C2；仅靠 gyro 相关线索勉强进 top-3（覆盖率 94%）。

> **批判**：以上根因中，②是**采集协议问题**（金标该采在真实场景 UI，或给第三方数据打标）、①④是**打分函数/特征标定问题**、③⑤是**特征抽取阈值问题**——**没有一条是"缺某类端侧信号"**。因此提升路由准确率主要靠"改协议 + 修特征 + 校准 LF/学习式路由"，而非新增传感器/字段。

## 7. 缺什么 + 改进优先级

| 优先级 | 改动 | 解决什么 | 说明 |
|---|---|---|---|
| **P0** | 让金标签落在**真实场景 UI** 上（受控任务改为在代表性真实 app/仿真场景界面完成），或为第三方数据引入可信（半监督/人工）场景标注 | 破除"金标在任务页、UI 不区分场景"（§6-②） | 当前 UI 类 LF 在金标数据上几乎失效；这是路由准确率的最大天花板 |
| **P0** | 多用户采集 | 训练可泛化的 `LearnedRouter`、并让认证置信度可定义 | 固定规则路由不需多用户，但学习式路由与认证需要 |
| **P1** | 修 `ui_surface_like` 抽取（视频/画布 surface 未被识别）、校准 `orient_landscape`（现反相）、复核 C6 运动阈值 | §6-③④⑤ 特征质量 | 属特征抽取层 bug/标定，改动局限、收益直接 |
| **P1** | 重校准 C2 打分（区分"偶发可滚动容器"与"持续滚动"，如提高 `scroll_count` 权重门槛、降低静态 `scrollable_count` 加分） | §6-① C2 吸收池 | 可先用 `--mapping`/温度/阈值网格在 validation 上调，勿在 test 上挑 |
| **P1** | 评估无障碍/输入通道能否补触控坐标或更细交互时序 | 更丰富的触控路由证据 | 现仅计数，`touch_rate` 够 C6 但不足更细区分 |
| **P2** | 把 `run_preprocess` 特征化 + `weak_label`/MoE 前向包成**在线推理端点** | 实现"上传即路由出分"的实时服务 | 当前 `app/` 为纯摄取（6 端点），路由仅离线 |
| **P2** | 报告用 **top-k 覆盖率**（而非只看 top-1）评估路由，并将 k 与 §RQ5 的 k* 联动 | 贴合 MoE"取 top-k 专家"语义 | top-3 覆盖 70.7% 说明正确专家常在候选内 |

## 8. 结论

- **Q1**：端侧 AccessibilityEvent + UI 快照 + 三通道 IMU + 触控时间戳，**在信号类型上足以驱动自动路由**（路由器全部输入均可派生、实跑已产出）；**缺的是"能区分场景的标注"与"特征抽取质量"，不是缺信号**：金标采在 App 任务页而非真实场景 UI、`ui_surface_like` 恒 0、`orient_landscape` 反相、触控无坐标。
- **Q2**：**服务端代码已实现自动场景路由**——固定规则路由（`weak_label`→`FixedRuleRouter`→`_topk_gate`）**零训练即可运行**并取 top-k 专家；学习式路由已建但**未训练**（需多用户）；且当前只在 `research/` **离线管线**中，摄取服务**无在线路由端点**。
- **Q3**：路由准确率 = `mean(weak_label_top1 == task_category)`（`run_preprocess.py:255`），**实测 top-1 = 23.3%**（top-2 64.7%、top-3 70.7%）；C1 好（0.72）、C0/C3/C4/C5/C6 top-1 极低，主因 **C2 吸收池 + 金标 UI 不真实 + 特征标定 bug**；学习式路由无准确率指标（需训练）。**该指标无循环论证，但金标是任务页上的"意图标签"**。
- **给评审者的一句话**：**自动路由"通"而"不准"**——链路与信号都在位，准确率瓶颈是**采集协议（金标该落在真实场景 UI）+ 少量特征抽取 bug + 标注函数校准**，修复路径明确且不需要新增端侧信号类型；但要评估路由对**认证**的增益，仍须先补齐多用户数据（另见 `数据可行性与实验需求满足度分析-2026-07-02.md`）。
