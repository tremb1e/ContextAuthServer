# HTTP 400 `schema_validation_failed` 根因分析与修复（2026-07-03）

> 现象：某次真机采集显示 **71 成功、35 失败**，失败提示 `HTTP 400 schema_validation_failed`。
> 本文给出根因、交叉定位过程、修复与端到端验证、数据损失评估与运维改进建议。

---

## 1. 结论速览

- **根因**：**线上部署的服务端容器镜像陈旧**。App 已于 2026-07-03（提交 `3327e8e`）把交互状态任务由 `C0..C6` 重构为 **`I0..I7`（8 类）**；服务端源码也已同步（提交 `6453b43`，2026-07-03 00:53，`TASK_CATEGORIES` 增加 `I0..I7`）。但**线上运行的镜像 `contextauth/server:deploy` 构建于 2026-07-02 14:04:39，早于该提交**，其 `TASK_CATEGORIES` 仍只有 `C0..C6`。
- **失败机理**：受控任务批次 `collection_source=BUILTIN_TASK`、`task_category∈{I0..I7}`。旧镜像在 `Batch.validate_task_label_contract` 中执行 `if task_category not in TASK_CATEGORIES: raise ValueError("invalid_task_category")` → `schema_validation_failed` → 隔离 → `HTTP 400`。而背景采集批次 `collection_source=THIRD_PARTY_APP`、`task_category=null`，**不进入** BUILTIN 校验分支，故正常落盘。
- **一句话**：**不是采集端数据坏了，是线上服务端没跟着源码更新**。71 个成功批次全部是第三方 App 背景采集，36 个失败批次全部是 App 内 `I0..I7` 受控任务金标签。
- **修复**：用当前源码（已含 `I0..I7`）**重建镜像并重新部署**（无需改任何 schema 代码）。已端到端验证：重部署后 `I0/I5/I7` 的 `BUILTIN_TASK` 与 `THIRD_PARTY_APP` 样例批次均返回 `stored:true / 200 / schema_ok=true`。

---

## 2. 现象与数据分区

对该设备 `36905bde…638` 在 `2026-07-03` 目录下的真实落盘做统计（Asia/Shanghai 时区）：

| 分区 | 数量 | `collection_source` | `task_category` | 时间窗 |
|---|---|---|---|---|
| 成功（`devices/…/*.json`） | **71** | 全部 `THIRD_PARTY_APP` | 全部 `null` | 12:42:24 – 12:50:54 |
| 隔离（`quarantine/…/*.json`，仅摘要） | **36** | （推定）`BUILTIN_TASK` | （推定）`I0..I7` | 12:38:46 – 12:41:40 |

关键观测：

1. **成功批次 100% 是 `THIRD_PARTY_APP`**（`task_category=null`），无一条受控任务金标签成功落盘。
2. **失败批次在时间上完整地早于成功批次**：12:38–12:41 是 App 内“按顺序完成 8 项受控任务”的引导采集段（`BUILTIN_TASK`），随后 12:42–12:50 用户切到第三方 App（小红书 / B 站 / 微信），背景采集（`THIRD_PARTY_APP`）全部成功。
3. `index/errors.jsonl` 中 36 条记录的 `reason` **全部是 `schema_validation_failed`**，无一条是解压 / 哈希 / base64 / `device_id` 类错误——即数据到达并成功解压、`json.loads` 成功，只是**结构契约**这一关没过。

> 计数口径说明：App 端提示“35 失败”，服务端隔离区实为 **36** 条。差 1 属客户端与服务端计数时点/口径差异（例如最后一个 400 批次仍在客户端重试队列、或客户端某一时刻的显示快照）。以服务端 `index/errors.jsonl`（36）为准，不影响结论。

## 3. 为什么隔离区看不到“具体是哪个字段错”

服务端隔离遵循隐私红线——**只存摘要、不留原文**。隔离文件形如：

```json
{"payload_summary": {"payload_sha256": "...", "payload_type": "json",
  "top_level_keys": ["app_package_name", ... , "task_category", ...]},
 "reason": "schema_validation_failed"}
```

`top_level_keys` 完整且正常（顶层字段一个不缺），说明**不是缺顶层字段**，而是**某个嵌套契约**未过；但摘要不含具体 pydantic 报错，也不含原文。因此根因定位不能只看隔离文件，而是用三方交叉验证。

---

## 4. 交叉定位（三方证据链）

**证据 A — 成功/失败分区**：成功全为 `THIRD_PARTY_APP`(null)、失败全为 `BUILTIN_TASK`(I0..I7)。这把嫌疑锁定在“**只对 BUILTIN_TASK 生效、且与 `task_category` 取值相关**”的契约上。

**证据 B — 线上容器实测 schema**：直接查线上容器里真正在跑的代码：

```
$ docker exec cca-deploy python -c "from app.schemas import TASK_CATEGORIES; print(sorted(TASK_CATEGORIES))"
['C0', 'C1', 'C2', 'C3', 'C4', 'C5', 'C6']     # ← 只有 C0..C6，没有 I0..I7！
```

而仓库源码 `app/schemas.py` 的 `TASK_CATEGORIES` 是 `{C0..C6, I0..I7}`。**线上 ≠ 源码。**

**证据 C — 镜像时间线**：

| 事件 | 时间 |
|---|---|
| 线上镜像 `contextauth/server:deploy`（旧）构建 | 2026-07-02 **14:04:39** |
| 服务端加入 `I0..I7` 的提交 `6453b43` | 2026-07-03 **00:53:00** |
| 服务端 HEAD `157bd1f` | 2026-07-03 01:18:01 |

**镜像构建早于 `I0..I7` 提交约 10 小时**——旧镜像里根本没有 `I0..I7`。这与证据 B 完全吻合。

**证据 D — 契约代码**（`app/schemas.py` `Batch.validate_task_label_contract`）：

```python
if self.collection_source == "BUILTIN_TASK":
    ...
    if self.task_category not in TASK_CATEGORIES:
        raise ValueError("invalid_task_category")   # ← I7 ∉ {C0..C6} 在旧镜像里必然触发
    ...
else:  # THIRD_PARTY_APP
    ...   # task_category 必须为 null，走这条分支，不校验类别集合
```

`ContextFeature.validate_category` 同理：`if task_category is not None and task_category not in TASK_CATEGORIES: raise invalid_task_category`——`THIRD_PARTY_APP` 的 feature `task_category=null`，被 `is not None` 短路放行；`BUILTIN_TASK` 的 feature `task_category=I*` 则被拒。

四条证据一致指向同一结论：**旧镜像只认 `C0..C6`，`I0..I7` 全被 `invalid_task_category` 拒绝**。

## 5. 排除“采集端二次 bug”

为确认修复只需重部署、采集端本身无隐藏契约问题，核对了 App 的批次构造（`core/JsonCodec.kt`）：

- 批次与每个 `context_feature` 的 8 个 task 字段**同源自同一个 `TaskCategory` 枚举**（`task_id/task_category = 枚举名`、`task_sequence = ordinal`、`task_name = taskNameEn`、`task_intuitive_description = intuitiveDescriptionEn`、`task_session_id` 同批一致）。因此服务端要求的 `context_feature.task_* == batch.task_*` 系列匹配契约**自动成立**。
- `task_id == task_category`（都取枚举名）成立；`task_sequence == int(task_id[1:])` 对 `I0..I7` 成立（`ordinal == int("I7"[1:]) == 7`）。
- `context_feature.event_id` 均取自本批 `context_events`（成功批次实测 `bad_ref=0`）。

**结论**：采集端 `BUILTIN_TASK` 路径与当前服务端契约完全一致，无二次 bug。修复只需让线上服务端跑上含 `I0..I7` 的当前源码。

### 6.1 顺带修复：样例工具违反 drop-all-text

端到端首测时 `tools/send_sample_batch.py` 生成的**样例**批次自身触发 `node_text_must_be_dropped`：其伪造的 UI 节点带占位 `"text":"确认"`、窗口标题 `"window_title_redacted":"<DROPPED>"`，违反“端侧丢弃全部文本”契约。已修正为 `text=None` / `window_title_redacted=None`（并补 `has_text` 存在位），使该冒烟工具符合隐私契约。

**注意**：这是**测试工具**的伪造数据缺陷，**与真机无关**——真机 `RedactionEngine` 恒把节点 `text/contentDescription/窗口标题` 置 `null`（成功落盘的 71 个真实批次即为证：无一条含文本原文）。该修复只为让冒烟工具可用、并与 App 一致把 I7 的 `task_name` 由 `Wrist rotation and motion canvas` 对齐为 `Wrist rotation`。

## 7. 数据损失评估

- 36 个 `I0..I7` 受控任务金标签批次**原文已不可恢复**：服务端隔离只存摘要不留原文；且返回的是 `HTTP 400`（客户端语义为“永久性拒绝”，通常不再重试），采集端本地大概率也已放弃。
- 因此本轮**没有任何 `I0..I7` 金标签数据落盘**（`by_category/` 下只有 2026-07-02 旧版 `C0..C6` 的软链，无 `I*`）。**修复后需重新完整采集一遍 8 项受控任务**（详见 `02-采集数据可行性…`）。

## 8. 运维改进建议（防止再次发生）

1. **契约即部署**：任何改动 `app/schemas.py`（尤其 `TASK_CATEGORIES` 等 ingest 契约）后，**必须重建并重部署镜像**——源码提交 ≠ 线上生效。本次问题的本质是“提交了 `6453b43` 但没重部署”。
2. **上线自检**：部署后跑一次 `tools/send_sample_batch.py --task-category I7`（BUILTIN）+ 一次 `THIRD_PARTY_APP`，确认 `stored:true` 再放量；或在 `/api/v1/config` 暴露一个 `schema/contract 版本`探针，App 端上传前比对。
3. **金标签监控**：给 `by_category/{I0..I7}` 落盘数加一个简单的每日计数告警——“受控任务金标签为 0”应立即报警（本次若有此告警可在采集当场发现）。
4. **隔离摘要可诊断性**：可考虑在隔离摘要中**附带脱敏后的 pydantic 错误定位**（如 `loc=('task_category',) msg=invalid_task_category`，不含原文），便于下次快速定位，同时不违反隐私红线。
