## 二、结论速览（三句话）

1. **失败根因不是采集端数据错误，而是线上服务端镜像未随源码更新**：App 已升级为 `I0..I7` 八任务金标签，但线上容器仍是 2026-07-02 构建的旧镜像（`TASK_CATEGORIES` 只含 `C0..C6`），因此每个带 `task_category=I0..I7` 的 `BUILTIN_TASK` 批次都触发 `invalid_task_category → schema_validation_failed → 隔离 → HTTP 400`；而 `THIRD_PARTY_APP` 批次 `task_category=null`，跳过该校验分支，故 71 个背景批次全部成功。
2. **修复即“用当前源码重建镜像并重部署”**（源码早已正确，无需改 schema）；重部署后线上 `TASK_CATEGORIES` 已含 `I0..I7`，`I0/I5/I7` 的 `BUILTIN_TASK` 与 `THIRD_PARTY_APP` 样例批次端到端返回 `stored:true / 200 / schema_ok=true`。
3. **数据可行性**：工程上修复后金标签采集可正常落盘，但本轮 `I0..I7` 金标签**已 0 条落盘（原文不可恢复，需重采）**；且即便重采，单设备 / 单用户仍无法定义冒充者与 `EER`，论文级实证仍需多用户研究（P0，与既往结论一致）。
