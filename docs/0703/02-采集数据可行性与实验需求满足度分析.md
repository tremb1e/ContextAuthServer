## 4. 建议（按优先级）

1. **立即重采一遍完整 8 项受控任务**（修复已上线）：采完后核对 `devices/…/{date}` 出现 8 类 `BUILTIN_TASK` 批次、且 `by_category/{I0..I7}` 均生成软链、`errors.jsonl` 无新增 `schema_validation_failed`。这是验证“工程链路真的通了”的最终一步。
2. **规划多用户研究**（P0，决定论文成立）：≥20–40 用户、每用户 ≥3–5 天、每场景 ≥2–3 session、覆盖多设备与多 App；这是唯一能支撑 `EER/FAR/FRR` 与 `matched_impostor` 的路径。合成数据仅验证流水线，不能替代该结论。
3. **口径对齐**：注意采集端已是 `I0..I7`（8 类）而离线实验层 `research/` 与 `exp_prompt` 仍是 `C0..C6`（7 专家）。服务端 ingest 已兼容两者；把离线实验迁到 8 类（或在研究层做 `I0..I7 → C0..C6` 的 canonical 映射）是独立后续工作（见服务端说明文档 §相关章节）。
4. **本轮 `THIRD_PARTY_APP` 数据的用途**：71 个背景批次可用于**流水线联测、特征/质量分布观察、无标签自监督预训练的可行性摸底**，但**不带场景金标签、且仍是单用户**，不能用于认证有效性结论。

---

## 5. 小结

本轮采集**不满足**后续认证实验的需求，原因有二且性质不同：

- **工程层面（已修复）**：镜像陈旧导致 `I0..I7` 金标签全被拒、0 条落盘——重采即可解决。
- **科学层面（仍存在）**：单设备/单用户无法定义冒充者与 `EER`——需多用户研究，非本次修复能解决。

因此下一步是“**先重采验证链路，再组织多用户研究**”。
