# Prompt Requirements Matrix

| ID | Area | Requirement | Acceptance |
|---|---|---|---|
| APP-01 | App shell | Kotlin + Jetpack Compose + Material 3, minSdk 26, targetSdk 35, ContextAuthLab branding | Debug APK builds and launches as ContextAuthLab |
| APP-02 | Consent | Significant disclosure before collection | User must consent before Home collection can start |
| APP-03 | Onboarding | Accessibility, battery optimization, notification permission flow | Automatic collection remains blocked until all three permissions pass |
| APP-04 | Home | `Collection Status` displays server connection state, automatic collection state, latest connectivity-test time, latest upload time, and latest server response; host-only server target, masked device ID, permissions, sensors, ClockSync, automatic collection readiness | No Home start/stop buttons; server connection chip triggers a fresh `/ready` readiness test; missing Accessibility, battery, and notification permissions are prompted one by one; no NTP host names in UI; no in-app rule version/hash display or "check rules" control; collection starts automatically when required permissions, valid research `device_id`, and screen/unlock state pass; network, ClockSync, and Wi-Fi failures do not block local sampling |
| APP-05 | Researcher settings | Hidden version/Home server gesture, editable server URL, readiness test, reset, clear queue, export diagnostics | URL save/reset requires confirmation where applicable and triggers ClockSync/config refresh; override banner appears when URL is changed |
| APP-06 | Built-in tasks | Eight C0-C7 adult-oriented task categories with sitting posture guidance and task labels | C1 scrollable protocol, C4 simulated phone settings, C5 fullscreen landscape blue-ball tapping challenge, C6 fullscreen local video player with overlaid pause/speed/seek/orientation controls that auto-hide after 2 seconds idle and reappear on tap, C7 wrist guide with stable forearm, wrist pivot, left-right swing, front-facing lateral translation with a fan-shaped wrist path, and forward-back flexion; BUILTIN_TASK batches include English task fields |
| APP-07 | Details | Runtime overview, app/device info, sensor rates, ScreenGate history, queue, upload, timing, local upload history, ClockSync | Details page has no one-click Diagnostics export button; local upload history shows file name, time, size, status, and response/error text |
| APP-08 | Device ID | HMAC-SHA256(serverStudySalt, ANDROID_ID), 64 lowercase hex, cached | No hardware IDs such as IMEI/serial/MAC/MediaDrm |
| APP-09 | Accessibility | Read-only node structure, foreground package/activity metadata, `viewIdResourceName`, input-method-visible state, and broad UI component attributes (no text content); a proactive foreground snapshot (`FOREGROUND_SNAPSHOT`) is pulled at each batch flush so the current foreground app's package and text-free UI structure are captured even with no reactive Accessibility events | No performAction, gestures, screenshots, auto-click, any displayed/entered text, per-key timestamps, key intervals, key hold durations, touch positions, or touch trajectories; service permission checks match the exact component and UI-window traversal is gated to active collection; the snapshot uses the same drop-all-text node walk and node-walk limits as reactive events; when the service is disconnected no snapshot is produced |
| APP-10 | Sensor collection | Accelerometer, gyroscope, magnetic field at fixed 100 Hz | registerListener uses 10,000 us sampling period; status displays measured acc/gyro/mag rates |
| APP-11 | ScreenGate | Stop collection while screen off or locked | Existing text-free samples/events are flushed or queued before pause, then the same task context can resume after unlock |
| APP-12 | Text handling and UI semantics | Drop password nodes and all displayed/entered text on-device; upload plaintext foreground `app_package_name`, Activity/ComponentName, `viewIdResourceName`, structure/state metadata, and `has_text`/`has_content_description` presence flags | No in-app text redaction engine and no cloud rule fetch; node `text`/`text_redacted`/`content_desc_redacted` and event `window_title_redacted` are always `null`; package-level app skipping is removed; serialized batches contain no displayed/entered text; server schema accepts the now-null text keys and requires `diagnostics.redaction_applied: true` |
| APP-15 | Internationalization | Chinese and English UI content selected by system language | Major participant-facing UI, tasks, notification, protocol, details, and dialogs have Chinese/English variants |
| APP-13 | Transport | UTF-8 JSON -> LZ4 frame -> SHA-256 envelope over HTTP(S), with non-empty `session_id` for task and foreground collection sessions | No AES/PBKDF2/AAD/public-key envelope protocol |
| APP-14 | Failure queue and upload history | filesDir/upload_queue, metadata DB, 200 MB FIFO, retry/dead letter, local upload history | Offline batches are persisted and replayable; uploads/replays/queued failures are recorded for Details history |
| APP-16 | Default foreground collection | No third-party collection allow/deny switch; when consent and required permissions pass, foreground app UI structure, global timing-only touch events, and sensors are collected together; collection is hosted by a process-scope coordinator and a foreground service so it survives the Activity being backgrounded/destroyed and captures other (third-party) apps during normal use | Every batch carries the foreground package, text-free UI structure nodes, and matching counts via a proactive `FOREGROUND_SNAPSHOT` whenever the Accessibility service is connected, so `context_events`/`context_features`/`app_package_name` are no longer event-only or `"unknown"`; sensor-only batches are still retained (e.g. when the service is disconnected); reactive Accessibility UI events and global touch timing are also included when present; the foreground service restarts third-party collection on start only when nothing is already running so an in-progress builtin task is never overridden. This holds after the 2026-06-11 capture fix; the earlier on-device dataset was sensor-only on Android 11+ because a Service-context `getDisplay()` call threw and silently dropped UI capture (see `docs/fixes/2026-06-11-foreground-ui-capture-fix.md`) |
| SRV-01 | Server API | Pure FastAPI API: health/config/rules/ingest/metrics | Config exposes China-region ClockSync metadata; `/api/v1/rules` is initialized from `SERVER_RULES_FILE`; no templates/static/dashboard routes |
| SRV-02 | Ingest | Validate envelope, hash, LZ4, JSON, Pydantic schema | Valid sample batch returns stored=true |
| SRV-03 | Privacy checks | Validate schema-level privacy invariants after server secondary scan removal | Reject editable raw text, password-node presence, missing redaction marker, invalid task contract, orphan context-feature event IDs, diagnostics count mismatches, or bad envelope without logging raw payload |
| SRV-04 | Storage | Store by device_id/date, meta, indexes, by_category, quarantine | Safe path resolution prevents traversal; repeated identical batch IDs are idempotent, conflicting duplicates are rejected without overwrite |
| SRV-05 | Logging/metrics | JSON lines logs and Prometheus metrics | No full device_id, raw payload, salt, key, or plain IP |
| DEV-01 | Docker | Server image plus Android APK artifact image; server runs non-root with bind-mounted data/logs | In `ContextAuthLabServer`, `docker build -t contextauthlab/server:latest .` and `tools/test_docker_deployment.sh` pass; in sibling `ContextAuthLabApp`, `make build-app-image` passes after the debug APK exists |
| DEV-02 | Tests | Android unit, APK build, server pytest, E2E, Docker smoke | Commands in README and Makefile pass in this environment |

Explicitly out of scope: authentication model, context routing model, MoE, training, inference, hidden collection, screenshots, automatic control, gRPC/proto, AES content encryption, anomaly detection, Dashboard/Web UI.
