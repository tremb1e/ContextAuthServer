### 1.3 「满足」的两层定义（结论指针）

直接问「数据能否满足实验」会误导，须拆两层分别判定：

- **层 A（工程 / 流水线可复现性）**：数据能否被实验层原样消费、端到端跑出 windows / datasets / 训练 smoke / 报告，且不违反隐私与去泄漏契约。
- **层 B（科学 / 认证结论有效性）**：数据能否支撑「按场景路由 → 提升持续认证准确率」这一命题的**可证伪实证**（真实 EER/FAR/FRR、跨用户 impostor、无泄漏协议、按用户统计）。

> **提前给出结论指针**：**层 A —— 满足（数据格式合规、已被摄取校验 `ok`、65 批全部成功载入、实跑产出 316 窗 / 204 维无泄漏特征）；层 B —— 不满足（单用户 → 0 冒充者对，每场景单 session、单日，金标窗口经重叠放大后独立样本仅约 22 个）**。二者都要交付、不可混淆。

## 2. 结论速览（一页纸）

1. **整体满足度（一句话）**：层 A **满足**（工程可端到端复现）；层 B **不满足**（认证有效性结论不可达）。当前数据是合格的**流水线验证 / 开发数据集**，不是合格的**论文实证数据集**。
2. **模型训练 部分满足**：M0/M7 可在真实数据上跑通 1-epoch smoke（204 维无泄漏特征就位），但因单用户无跨身份负样本，度量学习/对比损失退化、user 分类头塌为 1 类——**训练可运行、不可学到可评估的认证能力**。
3. **baseline 对比（M0–M10）不满足**：全部 11 个模型的落点都是真实 EER/AUROC；单用户下这些指标不可计算，M0–M10 之间**无法做有意义比较**（仅能比参数量/延迟等开销量）。
4. **消融 部分满足**：top-k 1..7 的**开销侧**（激活专家数/延迟/FLOPs/参数量）、单通道特征重算、隐私逐级粗化、弱标签置信筛选**机制上都能跑**；但每项的**准确率侧结论**均需多用户 EER，**不可得**。
5. **评估协议 不满足**：`leave_session_out` 因每场景恰 1 session 退化为 leave-one-class-out；`leave_day_out` 因单日不可能；`leave_app_out` 因金标全在单包且第三方无标签而不成立；`matched_impostor` 因单用户**冒充者对数 = 0**（实跑证实）。
6. **统计 不满足**：按用户分层 bootstrap 的重采样单位是「用户」，仅 1 个用户 → 95% CI、paired delta、Holm、per-user EER、heldout-user 主结论**全部无输入**。
7. **最大缺口 / 【P0】**：**单用户、0 冒充者**（认证任务无法定义）+ **每场景单 session、单日**（无泄漏评估无法构造）。
8. **循环论证陷阱**：`estimated_context_category == task_category` 对全部 455 条 builtin 特征 100% 恒等（实测），是恒等式非预测力，**不得作为性能证据**；已被实验层列为泄漏列强制剔除。
9. **弱监督信号偏弱**：实跑 top1 弱标签与金标一致率仅 **23.3%**，弱标签分布严重偏斜（C2 209 / C4 4），M7 的 KL 弱监督在本数据上**信号质量不足**。
10. **相较旧采集的进步（真实）**：8→7 场景已落地（`by_category/C0…C6`）、批次 42→65、**触控从全 0 变为 607 条起止时间戳**、隐私红线更干净、第三方 app 更丰富（5 包）；但**单用户/单 session/单日三大硬约束仍未突破**。
11. **最小必要改动 / 【P0】**：多用户（≥20–40）× 多日（≥3–5）× 每场景多 session（≥2–3）× 多设备多 app 的真实用户研究——不补则层 B 命题永不可证伪。
12. **可证伪锚点**：见 §1.4——当前数据不能完成该检验。

## 3. 关键警示（务必先读）

1. **单用户 = 只能做「上下文路由」原型，做不了真正的「认证」。** 磁盘仅 1 个物化设备（`36905bde…`）；index 里出现的第二设备 `bf5e423a…` 只有 1 条 BUILTIN-C3 索引、**无落盘目录**，不可用作第二用户。没有他人窗口就无法定义 impostor/负样本，EER/FAR/FRR/AUROC 无从评估。**【P0】**
2. **每场景恰 1 个 session。** 实跑证实 7 个金标场景各只有 1 个 `session_id`（`distinct sessions per gold scene = {C0..C6: 1}`）。按 session 分组交叉验证退化为 leave-one-class-out；§8.2 要求的「enroll/query 取自不相交 session」在任何场景上都无法满足。**【P0】**
3. **不要把「流水线不报错」当「数据可用」。** `build_datasets --protocol leave_session_out` 在真实数据上**成功产出**数据集，`leakage_all_true=True`——但这是**空真通过**：`n_genuine_pairs=0 / n_impostor_pairs=0`（单用户下 impostor 池为空，`impostor_pool_disjoint` 对空列表 `all(...)` 返回 `True`，`impostors.py:71-77`）。**「无泄漏」标记为真恰恰是因为没有任何可评估的对**。
4. **不要把「100% 命中」当性能。** builtin 段 `estimated_context_category == task_category`（455/455，实测）是直抄 `taskCategory` 的恒等式，是**循环论证**，不是预测能力。
5. **必须剔除的标签泄漏 / 指纹列**（实验层已在 `research/__init__.py:35-40` 定义并强制）：`estimated_context_category`、`game_like_score`、`viewIdResourceName`、`coarse_orientation`。本数据这些列**全部在场**（`game_like_score` 恒非 0、max 0.8；landscape 35 条集中在 C5——方向与媒体场景纠缠）。
6. **行为生物特征当前主要靠三通道 IMU；触控只有节奏没有轨迹。** 触控新增 607 条，但 schema 仅存 `TOUCH_INTERACTION_START/END` 时间戳，**无 x/y 坐标 / 压力 / 面积 / 轨迹**（`app/schemas.py` 触控模型）。可支撑「触控密度」类弱标签证据（如 C6 有 362 次触控），但**不能替代触控轨迹生物特征**。
7. **单用户下 IMU 统计可能编码「本次握持习惯 / 本会话」而非「场景」。** 须用跨用户、跨会话消融区分——当前数据无法进行该区分。

## 6. 逐实验需求可满足性矩阵

> 判定基于 §4 实测 + §5 实跑。状态词：满足 / 部分满足 / 不满足。

### 6.1 认证任务与评估协议

| 需求 | 映射 | 数据前提 | 状态 | 依据 |
|---|---|---|---|---|
| genuine/impostor + EER/FAR/FRR/AUROC/PR-AUC + prototype-cosine 验证 | 核心认证 / RQ1–RQ7 | ≥2 用户 | **不满足** | 单用户；实跑 impostor 对 = 0 |
| `leave_session_out`（同 session 不跨集且每折覆盖 7 类） | 协议 | 每场景 ≥2 session | **不满足** | 每场景恰 1 session → 退化为 leave-one-class-out |
| `leave_day_out`（留后续日测时间漂移） | 协议 | ≥2 天 | **不满足** | 全部单日 2026-07-02，跨度仅 311.55 s |
| `leave_app_out`（留 app 桶证明不记包名） | 协议 / M3 / M8 / RQ6 | 多个「带金标且可定义 genuine/impostor」的 app 桶 | **不满足** | 金标全在 `com.contextauth` 单包；第三方 5 包无标签且同一用户 |
| `matched_impostor`（他用户匹配弱标签/包类采负样本） | 协议 | ≥2 用户构成池 | **不满足** | 仅 1 用户；实跑对数 = 0（`impostors.py:173-179`） |
| 组合最严格列 `leave_day_out ∩ leave_app_out` | USENIX 重点 | 多用户+多日+多标签 app | **不满足** | 三前提全缺 |
| enroll/query 跨 session 不相交（抗 EER 虚低） | §8.2 关键约束 | 每场景 ≥2 不相交 session（理想跨天） | **不满足** | 每场景 1 session 且单日，5 s/1 s 重叠窗只能同 session 取样 |

| 模型 | 数据前提 | 状态 | 依据 |
|---|---|---|---|
| M0 sensor_only_dense | 多用户 | **不满足** | IMU 充足可 smoke，但无跨用户负样本、无 EER 评估 |

## 7. 核心论断一：工程 / 流水线可复现性（层 A）—— 满足

**结论：作为实验层的验证 / 开发数据集，这份真实数据基本够用——格式合规、全部载入、端到端跑通、去泄漏契约通过。**

### 7.1 输入端与流水线契约就位

- `load_batches`（`loaders.py:120-150`）以松校验 `REQUIRED_BATCH_KEYS`（`loaders.py:32-49`）读取，真实 65 批**全部满足**必备键、无一丢弃（对照：合成目录 `data/paper/devices` 会因缺键被丢，二者不可混淆）。
- 窗口化 `make_windows`（`windowing.py:128-134`，默认 5 s/1 s、按 `(device_id, session_id)` 分组 `windowing.py:170`）在真实数据上产出 316 窗；特征层输出 204 维、`leakage_free=True`。
- 7 类任务契约（`task_id==task_category`、`task_sequence==int(task_id[1:])`，`schemas.py:325-330`）与 App 枚举 `TaskCategory C0..C6`（`Models.kt`）一致；真实 builtin 段 `by_category/C0…C6` 七类齐备。

### 7.2 去泄漏与隐私由代码强制（修正循环论证风险）

泄漏列在三处被强制排除，与实测「泄漏列在场」正好互补（数据里有、训练时丢）：

- 权威集合 `LEAKAGE_COLUMNS = {estimated_context_category, game_like_score, viewIdResourceName, coarse_orientation}`（`research/__init__.py:35-40`）。
- 特征层反泄漏断言（`feature_extractors.py:176-179`）、结构哈希显式排除 `viewIdResourceName` 及一切文本（`:579-606`）、IMU 派生横屏布尔 `orient_landscape` 合法替代端侧 `coarse_orientation`（`:453-455`）。
- 弱标注允许列表与泄漏集导入期断言不相交（`interaction_states.py:85`）；数据集构建再断言 `feature_columns ∩ LEAKAGE_COLUMNS = ∅`（`builders.py:243-244`）。
- 隐私红线在数据侧亦已满足（§4.5：text 全 null、password 0）。

### 7.3 可行性小结 + 可证伪点

层 A 满足。**可证伪点**：若在真实数据上跑 `run_preprocess → build_datasets → 训练 smoke → make_report` 任一环节因数据格式失败，则层 A 不成立——实测四环节均通过（316 窗 / 258-51-7 划分 / 204 维特征 / 无泄漏）。**唯一「跑通但退化」之处**：`build_datasets` 的 genuine/impostor 对为 0（属层 B，见 §8）。

## 8. 核心论断二：科学 / 认证结论有效性（层 B）—— 不满足

**结论：当前数据无法支撑「按场景路由 → 提升持续认证」的任何可证伪实证；认证指标在数据上不可定义。**

### 8.1 唯一致命项：单用户 → 0 冒充者（P0）

持续认证的正负样本 = 同用户 genuine vs 他用户 impostor。磁盘仅 1 用户，`sample_matched_impostors` 候选过滤 `pool_user != guser`（`impostors.py:173`）对单用户恒空、`relaxed_topk` 回退仍空（`:175-177`）→ 每个 genuine 窗产不出对 → 实跑 `n_impostor_pairs=0`。**EER/FAR/FRR/AUROC/time-to-detect 全部无从计算。** 正中 exp_prompt §18.1 的 P0 判定。

### 8.2 评估结构缺陷：每场景单 session + 单日（P0）

- `leave_session_out` 委托 `_assign_by_fraction`（`splits.py:89-144`）；单 session（n=1）命中 `n<=2` 退化分支（`:118-124`）。真实数据虽有 15 session（含第三方），但**每个金标场景恰 1 session** → 任何「同场景 enroll/query 跨 session」都不存在。
- 单日 → `leave_day_out`、组合最严格列不可能。
- **净效果**：连一个「无 session/day 泄漏」的**路由分类器**都难以严格评估，遑论认证。

### 8.3 样本规模与独立性：金标运动 ~128 s、独立窗 ~22（P0 级）

7 类金标运动合计 128.29 s，5 s/1 s 加窗后 133 个金标窗**共享 80% 样本**、独立样本仅约 22 个（C2 约 0–1、C4 约 1）。以「用户 / 用户×场景」为统计单位时，per-scene EER 等指标**无统计意义**；把重叠窗当独立样本会系统性高估性能（§18.3 明令禁止）。

### 8.4 弱监督信号偏弱

实跑弱标签 top1 与金标一致率 23.3%、分布偏斜（C2 209 / C4 4）。M7 的 `confidence(q)·KL(q‖p_router)` 弱监督在本数据上**监督质量不足**；叠加触控无坐标削弱 C6「高触控密度」判据、方向与 C5 纠缠，弱路由的可靠性进一步受限。

### 8.5 可行性小结

层 B 不满足。**这不是调参能解决的问题，而是数据构成问题**：缺第二用户、缺每场景第二 session、缺第二天。

## 11. 差距清单 + 补采优先级

| 优先级 | 改动 | 为哪一步 / 解决什么 | 说明 |
|---|---|---|---|
| **P0** | **多用户、多设备采集（≥20–40 用户）** | 认证任务的定义前提 | 无第二用户则 impostor/负样本不存在，EER/FAR/FRR 永不可评估；不补则层 B 命题不可证伪 |
| **P0** | **每场景 ≥2–3 个独立 session（首选跨天）** | `leave_session_out` / enroll-query 不相交 / §8.2 抗 EER 虚低 | 破除「每场景单 session」退化，使无泄漏评估与 prototype 验证成立 |

| **P3** | 文档-实现一致性收尾（如避免与合成目录混淆的路径说明） | 可维护性 | 明确 `deploy/data/paper` 为真实数据、`data/paper` 为合成冒烟数据 |

## 12. 结论

- **交付两个层次，不可混淆**：这份 2026-07-02 真实采集**可作为实验层的端到端工程验证 / 开发数据集**（层 A 满足：65 批全部载入、316 窗 / 204 维无泄漏特征、四环节跑通）；但**不能支撑论文的认证有效性结论**（层 B 不满足：单用户 → impostor 对实测为 0、每场景单 session、单日、金标独立窗 ~22、弱-金一致率 23.3%）。
- **给评审者的一句话**：**当前数据够跑通、不够下结论**——「按场景路由提升持续认证」的可证伪实证，必须等到 ≥20–40 用户 × ≥3–5 天 × 每场景 ≥2–3 session × 多设备多 app 的真实用户研究到位后才能开展；在此之前，M0–M10 / top-k / 消融 / 统计只能作**工程冒烟**，其准确率数字**不得写入论文结论**。
