## 二、三篇核心结论速览

**01 采集数据全景清单**
- app 采集 = **三轴运动传感器原始时序**（加速度计/陀螺仪/磁力计，~100 Hz）＋ **零文本的 UI 结构/状态/类型快照**（无障碍节点树）＋ **一份端侧算好的手工特征向量与启发式上下文标签**（`context_features`）。
- **刻意不采**：任何文本内容（邮箱/用户名/密码/正文一律丢弃，仅留 `has_text`/`has_content_description` 两个布尔）、精确触控（坐标/压力）。
- 传输：JSON → LZ4 压缩 → base64 信封 → SHA-256 校验 → HTTPS（`encryption="none"`，仅传输层 TLS）；`device_id` 为加盐 SHA-256 假名；server 端 pydantic 严格校验。

**02 数据可行性与目标达成分析**
- 把"目标"拆两层评判：
  - **层 A（字面设想，含"采集邮箱/用户名/密码文字"）**：**不满足，且不应满足** —— 与现行 drop-all-text 隐私策略冲突，且文本对路由几无必要、对认证非主流特征而隐私代价极高。
  - **层 B（真实科研目标：上下文路由 → 提升持续认证）**：**部分满足** —— 路由原型所需输入数据基本齐备，可起步；但**端到端"按场景认证提升准确率"当前不可达**。
- 不可达的硬约束（P0）：**单用户、0 冒充者样本**（认证任务无法定义）、**触控全空且 API 层无坐标**、**每类有效数据极少**（8 类合计约 88 s 运动数据）、**每类金标签仅来自单一 session**（连无泄漏的路由评估都做不出）。

**03 MoE 自动路由与专家模型训练方案**
- MoE 的 gating/router = 一个**上下文分类器**：输入「包名/Activity + UI 结构特征 + 派生特征（+ 对齐的短窗运动特征）」→ 输出 8 专家之一（先 hard routing，后 soft top-k gating）。
- 推荐 **C0–C7 ↔ 8 专家严格双射**：C0→IDLE_HOLDING、C1→STATIC_READING、C2→SCROLL_BROWSE、C3→TYPING、C4→FORM_FILLING、C5→TAP_NAVIGATION、C6→VIDEO_WATCHING、C7→GAME_OR_TILT（C5 改判 GAME_OR_TILT 作消融备选）。
- 路线：**LightGBM 表格基线**（少样本友好、可解释、对标现有启发式）→ DeepSets/Set-Transformer/小型 GNN 建模 UI 节点集合 → UI＋运动**中间融合**；训练用类权重/focal loss、数据增强、半监督伪标签；**双轨评估**（同类 leave-one-batch-out 报跨批泛化 + 分层 5 折但显式标注含会话内泄漏）；M1–M4 里程碑直到多用户数据下评估认证增益。

## 四、贯穿三篇的关键警示（务必先读）

1. **单用户 = 只能做"上下文路由"原型，不能做真正的"认证"。** 没有第二个用户/冒充者，认证的正负样本无法定义，EER/FAR/FRR 等指标无从评估。**【P0】**
2. **每类金标签仅 1 个 session。** "按 session 分组交叉验证"会退化为 leave-one-class-out —— 当前数据连无泄漏的**路由分类器**都难以严格评估。**【P0】**
3. **不要把"100% 命中"当性能。** 引导集里 `estimated_context_category == task_category`（348/348）是恒等式（直抄 `taskCategory.name`），是**循环论证**，不是预测能力。
4. **必须剔除的标签泄漏/指纹列**：`game_like_score`（用了 `taskCategory==C5`）、`estimated_context_category`（引导段=金标签）、`viewIdResourceName`（含包名、易指纹化）、`coarse_orientation`（与 C5/C6 纠缠）。
5. **行为生物特征当前只剩运动传感器**（触控全空且无坐标）；且运动统计在单用户下可能编码"该用户握持习惯/本次会话"而非"场景"，须做消融区分。
6. **文本对路由/认证都非必需**，现行 drop-all-text 更契合合规；设想 step2 的"采文本"应改为"采结构+行为+运动"。

## 五、后续工作（补采优先级，详见 02 §8 与 03 §10）

- **P0**：多用户、多设备采集（认证可行性的前提）；为每个上下文类别采集 ≥ 多个独立 session（破除单 session 泄漏）。
- **P1**：评估无障碍/输入通道能否补充触控坐标、压力或更细的交互时序；扩大每类样本时长与真实第三方 app 覆盖。
- **P2**：补采各方向下的多场景样本，解开"方向 ↔ 场景"纠缠；为 THIRD_PARTY 数据引入半监督/弱标签。
- **P3**：清理 server `default_rules.json` 与 app drop-all 策略之间的语义矛盾，使文档与实现一致。
