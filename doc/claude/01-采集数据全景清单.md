## 2. 采集与数据流总览

### 2.1 两种采集源（`collection_source`）

| 采集源 | 触发方式 | 是否携带任务标签 | 本数据集批数 |
|---|---|---|---|
| `BUILTIN_TASK` | 用户在 app 内做 C0–C7 引导任务，`startCollection(scope, category)` 传入 `TaskCategory`（`CollectionCoordinator.kt:188`） | **是**，`task_*` 全字段非空 | 20 |
| `THIRD_PARTY_APP` | 无引导（`category == null`），由前台 App 触发无障碍事件被动采集；`buildBatch` 据 `taskCategory == null` 判定来源（`CollectionCoordinator.kt:561`） | **否**，`task_*` 全字段必须为 null（server 强制） | 22 |

### 2.2 端到云数据流

```
┌─────────────────────────── Android 端（android-app/src） ───────────────────────────┐
│                                                                                      │
│  无障碍服务 ResearchAccessibilityService                                            │
│    onAccessibilityEvent → handleAccessibilityEvent                                   │
│      ├─ 遍历前台 App 窗口节点树 traverse()（深度≤14，节点≤320）                       │
│      │    每个节点 → RawNodeSnapshot（含 text/contentDescription，仅本地用）          │
│      │    → RedactionEngine.sanitizeNode() 端侧脱敏：                                  │
│      │         · password 节点 → 整节点丢弃（返回 null）                              │
│      │         · text / contentDescription → 仅算 has_text / has_content_description │
│      │           两个布尔后立即丢弃，绝不存储                                          │
│      │    → NodeSnapshot（纯结构）                                                    │
│      └─ 组装 ContextEventSnapshot → AccessibilityEventBus.emit()                     │
│                                                                                      │
│  运动传感器 SensorCollector（100Hz 注册三轴）                                        │
│      onSensorChanged → SensorSample（x/y/z/accuracy/时间戳）→ 内存 buffer            │
│                                                                                      │
│  触控 ResearchAccessibilityService.emitGlobalTouchEvent（仅 START/END 时间戳）       │
│                                                                                      │
│  CollectionCoordinator（每 5 秒 flushBatch）                                          │
│      drain 传感器/事件/触控 → ContextFeatureExtractor.extract() 派生特征+启发式标签   │
│      → buildBatch() 组装 Batch（追加一帧 FOREGROUND_SNAPSHOT）                        │
│      → JsonCodec.batchToJson()（手写 JSON 序列化，text 恒 null）                      │
│      → LZ4_FRAME 压缩 → SHA-256(压缩字节) → Base64 → PayloadEnvelope                  │
│      → Uploader.uploadOrQueue() POST /api/v1/ingest（失败入队重放）                   │
└──────────────────────────────────────────────────────────────────────────────────┘
                                       │  HTTPS / TLS（encryption="none"，无应用层加密）
                                       ▼
┌─────────────────────────────── Server（server/app，FastAPI） ───────────────────────┐
│  POST /api/v1/ingest（main.py:151）                                                  │
│    Envelope 校验 → Base64 解码 → SHA-256 比对（integrity.py）→ LZ4 解压               │
│    → Batch.model_validate()（schemas.py 严格 pydantic，密码节点/可编辑原文一律拒绝）  │
│    → DiskStore.store()：落盘 {batch_id}.json（明文 batch）+ {batch_id}.meta.json     │
│       （storage.py），按 device_id/日期/task_category 建目录与软链                    │
└──────────────────────────────────────────────────────────────────────────────────┘
```

### 2.3 批次切分与节奏

- **批时长**：固定 5 秒一批（`AppSettings.batchSeconds=5`，`SettingsStore.kt:50` `FIXED_BATCH_SECONDS=5`；`CollectionCoordinator.kt:224` `delay(batchSeconds*1000)`）。实测 `batch_duration_seconds` 取值 0–6 秒（含起止边界、屏幕门控提前 flush 等导致的短批）。
- **任务时长**：引导任务固定 30 秒（`FIXED_TASK_SECONDS=30`）。
- **屏幕门控**：熄屏/锁屏暂停并 flush 当前批（`CollectionCoordinator.kt:247,499`），解锁后可续采。
- **空批不发**：传感器、触控、上下文三者全空的批直接丢弃（`CollectionCoordinator.kt:363`）。

### 4.2 采样率、通道与数量统计

- **注册方式**：100Hz（`SAMPLING_PERIOD_US=10_000`），最大上报延迟 200ms（`MAX_REPORT_LATENCY_US=200_000`，`SamplingConfig.kt:5–6`；`SensorCollector.kt:116–125`）。实际有效率取 `min(100Hz, 传感器 minDelay 上限)`（`SensorCollector.kt:129–137`）。
- **三个通道**：加速度计、陀螺仪、磁力计；均通过 `getDefaultSensor` 注册（`SensorCollector.kt:64–66`）。
- **数量（全集 42 批，实测）**：

| 通道 | 全集样本数 | 单批样本数（min/mean/max） |
|---|---|---|
| ACCELEROMETER | 16 994 | 3 / 405 / 624 |
| GYROSCOPE | 16 969 | 7 / 404 / 624 |
| MAGNETIC_FIELD | 16 384 | 3 / 390 / 604 |
| **合计** | **50 347** | 单批三通道合计 13 / 1199 / 1852（中位 1521） |

> 说明：5 秒批 × 100Hz × 3 通道 ≈ 1500 条/批为典型量级；短批（屏幕门控、起止边界）会显著拉低下限。

### 4.3 坐标系与单位

- 坐标系遵循 Android `SensorEvent` 标准设备坐标系（屏幕正向时 x 右、y 上、z 出屏）。
- 单位：加速度计 m/s²（含重力）；陀螺仪 rad/s；磁力计 μT。
- app 不做任何重力分离、滤波或重采样——`x/y/z` 为**原始读数直传**。

### 5.2 关键事实：仅时间戳、无坐标、且本数据集全空

- **schema 中没有 x/y 坐标、没有压力、没有触点面积、没有手指数**——仅 4 个时间/类型字段。
- **本数据集 42 批 `touch_events` 全部为空（实测合计 0 条，0 批含触控）。**
- 原因（代码层面）：无障碍服务的触控事件来自 `AccessibilityEvent.TYPE_TOUCH_INTERACTION_START/END`（`ResearchAccessibilityService.kt:157–159`）。这类事件**仅在无障碍"触摸探索（Explore-by-Touch）"模式开启时**才由系统派发，且**本身就不携带触点坐标/压力**——Android 无障碍 API 出于隐私不向普通无障碍服务暴露原始 `MotionEvent` 的逐点坐标。普通使用（未开启触摸探索）下，系统不产生这两类事件，故本集恒为空。
- 含义：触控通道当前**几乎不可用作行为特征**，更像一个为兼容预留的占位通道。

### 6.2 节点采集与"前台归属"约束

- 仅采集**前台应用窗口**（`TYPE_APPLICATION`）下、且包名与前台一致的节点树（`ResearchAccessibilityService.kt:214–249`），避免抓到通知栏/输入法等无关窗口。

### 6.4 `bounds_grid` 量化方式（已读 `ResearchAccessibilityService.kt` 核实）

- 取 `AccessibilityNodeInfo.getBoundsInScreen(rect)` 得屏幕像素矩形，再对四个边各做整数除法 **`/24`**：`left=rect.left/24, top=rect.top/24, right=rect.right/24, bottom=rect.bottom/24`（`ResearchAccessibilityService.kt:120–125`）。

- **数据卫生提示**：离屏或未布局节点的 `getBoundsInScreen` 会返回极值矩形，`/24` 后出现哨兵值（实测 `left/top` 出现 `89478485`、`right/bottom` 出现 `-89478485`）。建模前应对 `bounds_grid` 做范围裁剪/异常过滤。

### 7.2 各 `*_like_score` 精确计算规则（`ContextFeatureExtractor.kt`）

- **`media_like_score`**（`:19`）：`node_class_histogram` 的键中**任一含 `Surface` 或 `Player`（忽略大小写）→ 0.8，否则 0.0**。
- **`list_like_score`**（`:20`）：`scrollable_count > 0 → 0.8`，否则 `0.1`。
- **`form_like_score`**（`:21–25`）：`editable_count ≥ 2 → 0.8`；否则 `editable_count == 1 且 clickable_count > 2 → 0.6`；否则 `0.1`。
- **`game_like_score`**（`:26–30`）：`task_category == C5 → 0.8`；否则 `media_like_score > 0.5 且 clickable_count ≤ 1 → 0.4`；否则 `0.1`。

### 7.3 `estimated_context_category` 启发式（`ContextFeatureExtractor.kt:31–39`）

按**优先级短路**判定：

1. **有引导标签** `taskCategory != null` → 直接用 `taskCategory.name`（C0–C7）；
2. 否则 `editable_count ≥ 2` → `C4`；
3. 否则 `input_method_visible` 或 `event_type=="TYPE_VIEW_TEXT_CHANGED"` 或 `editable_count == 1` → `C3`；
4. 否则 `event_type=="TYPE_VIEW_SCROLLED"` 或 `scrollable_count > 0` → `C2`；
5. 否则 `media_like_score > 0.5` → `C6`；
6. 否则 `clickable_count > 4` → `C4`；
7. 否则 → `UNKNOWN`。

> 即：**无引导（第三方 App）时启发式只可能产出 C2/C3/C4/C6/UNKNOWN**，永远不会自动产出 C0/C1/C5/C7（这些只能来自引导标签）。本集 C6 估计（47 条）经核验**全部来自引导标签**——因 `media_like_score` 恒 0，第 5 条规则在本集从未触发。

**`estimated_context_category` 全集分布（实测）**：C2 290、C4 152、C1 65、C6 47、C7 44、C5 26、C3 24、C0 8、UNKNOWN 14（合计 670）。

## 9. 设备 / 会话 / 同意 / 规则 / 完整性 / 传输元数据

### 9.1 设备假名化 `device_id`

- **不是简单加盐 SHA256，而是 HMAC-SHA256**：`HmacSHA256(key=serverStudySalt, msg=ANDROID_ID)` 的十六进制（`DeviceIdProvider.kt:47–51`）。盐为 `"Continuous_Authentication"`（`Models.kt:6`；server `config.py:10` 同值）。
- 64 hex 串，本集单一设备 `36905bdefcce…3638`；app 与 server 双侧均以 `^[a-f0-9]{64}$` 校验（`DeviceIdProvider.kt:33`、`schemas.py:9,95–100,292–297`），且 server 校验 `envelope.device_id == batch.device_id`（`main.py:237`）。
- 设备 ID 持久化在 `EncryptedSharedPreferences`（`DeviceIdProvider.kt:16–23`）。

### 9.2 会话 / 同意

- `session_id`：引导任务下取 `task_session_id`，无引导取采集会话 UUID（`CollectionCoordinator.kt:576`）；本集 18 个会话。
- `consent_version`：硬编码 `"1"`（`JsonCodec.kt:50`）。同意在端侧 `consentGranted` 门控采集（`CollectionCoordinator.kt:313`）。

### 9.3 规则版本 / 哈希（重要说明）

- 序列化恒为 `rule_version="1"`、`rule_hash`=端侧持久化值（`JsonCodec.kt:48–49`）。当前 HEAD 已**移除云端规则拉取**（删除 `RuleUpdateClient`），`SettingsStore` 不再有 `setRule()`，仅从 SharedPreferences 读取 `rule_hash`（缺省为 64 个 0 的基线，`SettingsStore.kt:46`；`Models.kt:228–229`）。
- **但本数据集 42 批的 `rule_hash` 全为非零的 `c6717ff10c…e51e`**（实测），而非基线 0。核实结论：该值是**早期含云端规则拉取的旧构建持久化下来的残留哈希**——升级到 drop-all 构建后，序列化仍读取 SharedPreferences 中旧的持久化 `rule_hash`，故继续上报。佐证：该哈希既不等于零基线，也不等于 server 当前 `default_rules.json` 重算的哈希（`6aab0e60…`）。
- **这不影响数据语义**：本集 payload 已是新隐私模型（`redaction_summary` 用新键、文本全 null、含 `has_text`/`has_content_description`），脱敏行为与 HEAD 一致；`rule_hash` 仅是一个"出身标记"，建模时不应据其推断脱敏策略。详见 §10。

### 9.4 传输与完整性

- 链路：`手写 JSON → LZ4 Frame 压缩 → SHA-256(压缩字节) → Base64 → PayloadEnvelope(JSON) → POST /api/v1/ingest`（`JsonCodec.kt:75–101,114–121`；`Uploader.kt:124–136`）。
- `payload_sha256_hex`：对**压缩后字节**取 SHA-256；server 用 `hmac.compare_digest` 防篡改比对（`integrity.py:20–21`；`main.py:184`）。
- 压缩效果（实测 42 批）：压缩 4 220–213 471 B（均值 85 330），解压 23 618–2 056 982 B（均值 720 986），**平均压缩比 ≈ 8.1×**。
- **`encryption="none"` 的真实含义**：应用层不加密，机密性完全依赖传输层 HTTPS/TLS（默认 `https://cca.macrz.com`，`Models.kt:5`）。server 端也声明无解密（`main.py:28` 注释"performs no decryption"）。
- 失败重传：本地队列 + 全抖动指数退避，上限 200MB、最多 20 次重试后入死信（`Uploader.kt:199–223`）。

## 10. 端侧隐私脱敏策略

核心实现 `RedactionEngine.kt`（类注释 `:45–50` 明确"All displayed/entered TEXT content is dropped on-device"）。

1. **drop-all-text（丢弃全部文本）**：每个原始节点的 `text` 与 `contentDescription` 仅用于计算 `has_text` / `has_content_description` 两个布尔，随即丢弃，绝不写入 `NodeSnapshot` 也绝不序列化（`RedactionEngine.kt:57–84`；`Models.kt:134–137` 注释）。窗口标题同样永久丢弃（`JsonCodec.kt:150–152`）。
2. **密码节点整节点丢弃**：`raw.password == true` 时返回 null，节点本身不进入快照，且遍历不下钻其子树（`RedactionEngine.kt:53–56`；`ResearchAccessibilityService.kt:130`）。序列化出的 `password` 恒 false。
3. **`bounds_grid` 空间粗化**：像素 ÷24 网格（见 §6.4）。
4. **`redaction_summary` 计数含义**（`RedactionEngine.kt:36–42`）——**这是端侧"丢了多少"的实时计数，并非全零占位**：

| 键 | 含义 | 全集累计（实测） |
|---|---|---|
| `dropped_password_nodes` | 丢弃的密码节点数 | 0 |
| `dropped_editable_texts` | 丢弃的可编辑控件文本数 | 0 |
| `dropped_text_nodes` | 丢弃的（非编辑）文本节点数 | **15 361** |
| `dropped_content_descriptions` | 丢弃的内容描述数 | **4 467** |
| `dropped_window_titles` | 丢弃的窗口标题数 | 0 |

> `dropped_text_nodes`(15361) 与节点级 `has_text=true`(15361) 完全一致、`dropped_content_descriptions`(4467) 与 `has_content_description=true`(4467) 完全一致——交叉印证脱敏计数真实有效。

5. **server 端残留脱敏字段**：server schema 仍保留 `text/text_redacted/content_desc_redacted`（`schemas.py:149–151`，均 `str|None`）与云端规则结构 `UiRedactionRule`/`RulesResponse`（`schemas.py:42–80`）、`default_rules.json`（仍含 email/phone/url 等 7 条 REDACT 规则）。但**app 端已不消费这些规则**（无任何 `/api/v1/rules` 调用），故它们处于 vestigial（遗留兼容）状态：app 实际策略恒为 drop-all。

## 11. server 端校验契约要点（"什么数据被允许"）

`schemas.py` 关键约束（理解数据合法边界）：

- **Envelope**（`:83–117`）：`algorithm` 必须 `"LZ4_FRAME+JSON"`；`device_id`/`payload_sha256_hex`/`rule_hash` 必须 64 hex；`batch_id` 必须合法 UUID。
- **NodeSnapshot**（`:129–161`，`extra="allow"`）：**密码节点直接拒收**（`password=true → password_node_must_be_dropped`）；**可编辑控件若带非空原文（且不是占位符）直接拒收**（`editable_text_must_be_dropped`）。app 的 `text:null` 恒满足。
- **SensorSample**（`:164–173`，`extra="forbid"`）：`sensor_type` 仅三枚举；时间戳非负；`accuracy` 可空。
- **TouchEvent**（`:176–191`）：仅 7 种 `event_type` 枚举 + 三个非负时间戳，**无坐标字段可填**。
- **Batch 级契约**（`:309–375`）：
  - `started ≤ ended`；
  - `BUILTIN_TASK` 必须 8 个 `task_*` 全非空，且 `task_id==task_category`、`task_sequence==int(task_id[1:])`；
  - `THIRD_PARTY_APP` 必须 8 个 `task_*` 全为 null（`third_party_task_fields_must_be_null`）；
  - 每条 `context_feature` 的 `collection_source/task_*` 必须与批一致，且其 `event_id` 必须能在 `context_events` 中找到；
  - `diagnostics` 的 `sensor_sample_count/context_event_count/touch_event_count/sampling_rate_hz` 必须与实际列表长度/采样率一致（四项交叉校验）。
- **入库流程**（`main.py:151–291`）：校验失败 → 隔离（quarantine）或拒绝；落盘明文 batch + meta，meta 中 `compressed_payload_omitted=true`（不留压缩体）、`schema_validation_result="ok"`（`storage.py:135–145`）。本集 42/42 校验 ok。

## 14. 小结：当前 app 到底采了什么、没采什么

**采了**：
- 三轴运动传感器（加速度计 / 陀螺仪 / 磁力计）100Hz 原始时序（x/y/z/accuracy/双时间戳），是当前**信息量最大、最连续**的模态；
- 当前前台 App 所见 UI 的**纯结构骨架**：控件类型（class_name）、开发者 viewId、粗量化布局位置（bounds_grid ÷24）、树深度/子节点数、十余个交互状态布尔、可执行动作摘要，以及"是否曾有文本/内容描述"两个存在性布尔；
- 前台上下文：包名、Activity/组件名、屏幕粗方向、输入法是否可见、无障碍事件类型流；
- 一份端侧算好的**手工特征向量 + 启发式上下文类别标签**（editable/scrollable/clickable 计数、四个 *_like_score、类名直方图、estimated_context_category）；
- 引导任务（C0–C7）的标签与计时；以及设备假名、会话、规则版本/哈希、完整性与压缩元数据。

**没采（或不可用）**：
- **任何 UI 文字内容**——显示文本、输入内容、内容描述、窗口标题全部端侧丢弃，仅留布尔存在性；
- **触控坐标 / 压力 / 轨迹**——触控通道只有 START/END 时间戳且本数据集全空（无障碍 API 不暴露原始触点，且需触摸探索模式才有事件）；
- 精确像素坐标（被 ÷24 粗化）、密码控件（整节点丢弃）、聊天记录等正文。

> 一句话：**当前 app 采的是"运动传感器原始时序 + 零文本的 UI 结构/状态/类型快照 + 端侧特征与启发式标签"，刻意不采任何文本与精确触控，文本仅以 has_text/has_content_description 两个布尔留痕。**
