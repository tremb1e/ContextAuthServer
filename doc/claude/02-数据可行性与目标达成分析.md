## 1. 文档元信息与方法论

### 1.1 评估对象与边界

本文回答一个问题：**当前实际采集到的数据，是否满足研究者的 idea？后续能否实现该 idea 的目的？**

研究者 idea 的 5 步为：(1) 无障碍取屏幕 UI 组件；(2) 端侧"可云端更新的固定策略"脱敏，**默认采集 UI 文字（邮箱/用户名/密码等）+ 组件类型 + 深度**，但不采正文；(3) 加密压缩上传；(4) server 按"包名 + 脱敏 UI"用 MoE 路由到 8 专家之一（IDLE_HOLDING / TYPING / SCROLL_BROWSE / STATIC_READING / VIDEO_WATCHING / TAP_NAVIGATION / GAME_OR_TILT / FORM_FILLING）；(5) 在被路由的专家上做认证以提升持续认证准确率。其中 step4、step5 **未实现**，当前 app 只采集上传、server 只存储。

### 1.2 "满足"的两层定义（关键方法论）

直接拿"是否满足 idea"提问会得到误导性答案，因为 idea 的**字面措辞**与其**真实科研目标**之间存在张力。故本文把"满足"拆成两层，分别评估：

- **层 A（字面 idea）**：照搬 5 步原文，**包括 step2 的"采集邮箱/用户名/密码文字内容"**。这是研究者写下的字面需求。
- **层 B（真实科研目标）**：以"**上下文感知的 MoE 路由 → 在专属专家上提升持续认证准确率**"为终极目的，step2 的具体采集项只是手段，可被替代。

提前给出结论指针：**层 A 不满足且不应满足（端侧已演进为 drop-all-text，与"采文字"直接冲突，且采文字对目标贡献低、隐私代价高）；层 B 部分满足——"上下文路由"原型数据基本够用，但"端到端认证提升"因单用户、无冒充者、触控为空而当前不可达。**

### 1.3 判据来源

- **代码**：`android-app/`（采集/脱敏/特征）、`server/app/`（schema/规则/存储）。
- **数据**：`data/testdata/2026-06-16/` 全 42 个明文 batch + 42 个 meta，全部经本文重新统计（非引用简报）。
- **可证伪性**：每条结论尽量给出"若 X 则结论改变"的反驳条件。

## 4. 核心论断一：对"路由（MoE gating）"是否够用？

**结论：作为原型，基本够用且已有可工作的弱版本；但"已存在 estimated_context_category 且 100% 命中"不能作为可行性证据（属循环论证），真正的无引导路由能力被严重高估。**

### 4.1 输入端数据齐备

路由的判别输入在当前数据中齐全：
- **包名**：每批 `app_package_name`（`JsonCodec.kt:34`）。
- **UI 结构**：`class_name` 直方图（TextView 15846 / View 8609 / Button 2442 / ...）、`scrollable`(2.14%)/`editable`(0.19%)/`clickable`(22.6%) 计数、`viewIdResourceName`（7651 节点有值，含 `recyclerview`/`ViewPager`/`SeekBar` 等强语义控件）。
- **短窗传感器**：~5s 批内每通道 ~500 样本，足以刻画"静止 vs 滑动 vs 转腕"的运动包络。
- **方向**：`coarse_orientation`（portrait 619 / landscape 51）。

从特征工程角度，"区分 8 类上下文"是一个**结构 + 行为高度可分**的问题，原型可行性较高。

### 4.2 但"100% 命中"是循环论证，不能采信（修正 C）

简报与聚合统计会显示"引导批 estimated == task 的命中率 = 348/348 = 100%"。**这是一个陷阱**：
- `ContextFeatureExtractor.kt:31-32`：`val estimated = when { taskCategory != null -> taskCategory.name; ... }`。即**只要有金标签，"估计值"就是直接抄写金标签**，根本没跑启发式分支。
- 因此 100% 命中是**恒等式**，不是预测性能。它只证明"代码把标签复制对了"，**完全不能证明启发式或任何模型在无标签时能分对 8 类**。
- 更糟：`game_like_score`（`:26-30`）显式使用 `taskCategory == TaskCategory.C5`，`estimated_context_category` 本身也内嵌标签——这两个特征在引导数据里**直接编码了标签，属标签泄漏**。任何"用现有特征即可分场景"的乐观结论，**必须先剔除 `estimated_context_category` 与 `game_like_score` 两列**后再评估，否则是自欺。

### 4.3 真·启发式只覆盖 3 类，偏置比预想更严重（修正 A）

把 `estimated_context_category` 按 `collection_source` 拆开，只看**无引导的第三方批**（启发式真正生效的场景，322 条特征）：

| estimated | THIRD_PARTY 计数 |
|---|---|
| C2（滑动） | 256（**79.5%**） |
| C4（多控件/表单） | 50 |
| C3（输入） | 2 |
| UNKNOWN | 14 |
| C0/C1/C5/C6/C7 | **0** |

即**现成启发式实际只能弱标注 C2/C3/C4 三类**（且严重塌缩到 C2）。聚合分布里出现的 C0/C1/C5/C6/C7 **全部来自 20 个引导批的金标签直通**（`ContextFeatureExtractor.kt:31-32` 短路）。特别地：
- **C6（视频）从未被启发式产出**——因为 `media_like_score` 在全集 670 条特征中**恒为 0**（`ContextFeatureExtractor.kt:19` 依赖 `class_name` 含 "Surface"/"Player"，本集视频走系统播放器，无障碍树未暴露该类名），`:29` 与 `:36` 两条依赖 media 的规则从未触发。
- **C0（静止）/C1（阅读）/C5（点击挑战）/C7（转腕）无法靠启发式弱标注**：C0/C1/C7 是纯运动语义、UI 上无区分；C5 在无障碍层与普通点击页无差别。

> **含义**：若要给 22 个无标签第三方批做弱/半监督标注以扩大训练集，**现成启发式只能覆盖滑动/表单/输入**，其余 4–5 类必须另想办法（运动信号聚类、引导任务迁移等）。弱标签偏置是 step4 落地的实质障碍，而非简报暗示的"已基本可用"。

### 4.4 路由可行性小结

- **可行性**：作为**上下文识别原型**，包名 + 结构 + 短窗传感器在方法上足以训练一个 8 类路由器，本数据的输入端齐备。
- **当前证据强度**：**弱**。100% 命中是恒等式；无引导启发式只覆盖 3 类；存在标签泄漏列。
- **可证伪点**：若剔除 `estimated_context_category`/`game_like_score` 后，用剩余结构 + 传感器特征在**会话不泄漏**的划分下仍能把 8 类分到显著高于随机（1/8=12.5%），则路由可行性被正面证实。**当前数据无法完成这个检验**（见 §5.3）。

## 5. 核心论断二：对"持续认证本体"是否够用？

**结论：当前数据明确不足以实现 step5。** 三条硬约束任一都足以否决，叠加后无回旋余地。

### 5.1 唯一有效生物特征是运动传感器，但规模与多样性严重不足

- 运动传感器是当前**唯一**成规模、具判别潜力的生物特征信号（质量好：~103Hz、真重力、无丢点）。
- 但**总量极小**：全集 wall-clock 跨度仅 **167.8s**，批时长合计 **176.3s**。按金标签类拆分，**每类可用传感器时长**为：

| 类 | C0 | C1 | C2 | C3 | C4 | C5 | C6 | C7 |
|---|---|---|---|---|---|---|---|---|
| 传感器秒数 | 4.3 | 18.8 | 7.2 | 4.5 | 18.7 | 9.7 | 11.7 | 13.8 |
| accel 样本 | 376 | 1851 | 692 | 383 | 1786 | 964 | 1135 | 1400 |

8 类合计约 **88s** 有标签运动数据。用 ~4–19 秒/类的单人数据训练"按场景的认证专家"在统计上不可行——连单个专家的本人模型都难以稳健估计，遑论 8 个。

### 5.2 触控为空且 schema 无坐标/压力——且是 API 根本限制

- 42 批 **0 条触控**；`TouchEvent`（`schemas.py:176-191`，`extra="forbid"`）只有事件类型 + 时间戳。
- **更深层**：app 能拿到的触控只有 `TYPE_TOUCH_INTERACTION_START/END`（`ResearchAccessibilityService.kt:157-175`），这是 AccessibilityService API 在**不开启全局手势拦截/不做无障碍 AccessibilityGestureEvent** 前提下的天花板——**无障碍框架本就不提供逐点坐标/压力/面积**。因此"补触控坐标"不是改 schema 能解决的，需评估替代路径（见 §8）。触控作为生物特征，当前**贡献 ≈ 0**。

### 5.3 单用户 = 认证任务无法定义；且单 session = 路由都无法无泄漏评估（修正 B，P0）

- **单用户（device_id=1）**：无第二人、无冒充者，持续认证的"负类"不存在，**认证任务无法构造**。这是 step5 的根本否决项。
- **每类仅 1 个 session（P0 硬约束）**：金标签 C0–C7 **各只来自唯一一个 task_session_id**。后果是——为防"同会话样本同时进训练/测试"导致的乐观偏差，标准做法是按 session 分组交叉验证（GroupKFold / leave-session-out）。但**每类只有 1 个 session**，意味着：
  - 要"会话不泄漏"，必须把某类的整个 session 留作测试 → 该折训练集**完全没有该类** → 退化为 leave-one-class-out，**无法在同一折里既不泄漏又覆盖全 8 类**。
  - 反之，要每折覆盖 8 类，就必须在同一 session 内切分 → **必然发生会话内泄漏**，评估结果虚高。
  - **结论**：当前数据连一个**严格无泄漏的路由分类器评估**都做不出来，认证更无从谈起。这是 P0 级数据结构缺陷，比"样本少"更致命。
- **方向几乎与 {C5,C6} 绑定**：51 个 landscape 事件中 C6=32、C5=17、第三方=2。横屏是 C5（点击挑战）与 C6（视频）的近专属信号；在引导数据里方向与标签强共变，构成又一处"伪可分"风险——分类器可能只是学会了"横屏→C5/C6"，而非真正的上下文/身份判别。

### 5.4 认证可行性小结

step5 当前**不满足**，且缺口是结构性的（缺用户维度），非靠"多采点"可弥补：必须改变采集设计（多用户 + 多 session + 冒充协议）。在补齐前，任何"按场景认证提升准确率"的声明都不可证伪、不可训练、不可评估。

## 6. "采文本"到底需不需要？（批判性立场）

idea-step2 想采"邮箱/用户名/密码等文字"。本文给出**明确立场：不需要，且现有 drop-all-text 是更优解**。分两个目的论证：

### 6.1 对"路由"——几乎无必要

- 区分 8 类上下文靠的是**结构 + 行为**：有无 `EditText`/键盘可见（输入）、`scrollable` + RecyclerView（滑动）、播放器/全屏横屏（视频）、多 Tab/滑块/复选框（多控件）、运动包络（静止/转腕/点击）。这些在当前 0 文本数据里**已全部可得**。
- 文本内容（某个邮箱的具体字符）对"这是输入页还是滑动页"**没有增量信息**——`has_text` 这个布尔已经捕获了"此处有文字"的结构事实，足够路由用。
- 实证：§4 显示路由的瓶颈在**弱标签覆盖与标签泄漏**，不在"缺文本"。补文本无助于解决这些瓶颈。

### 6.2 对"认证"——非主流生物特征且隐私代价极高

- 行为生物特征认证主流依赖**运动 / 触控动力学 / 击键时序**，而非"用户输入的文字内容本身"。知道用户输入了 `zhang@x.com` 对"这是不是本人"几乎无判别力（任何人都能打出同样的字）。真正有判别力的是**击键间隔/按压时长**——而这些 README 第 26 行明确不采集。
- 采集邮箱/用户名/**密码明文**会引入极端隐私与合规风险（明文凭据离开设备），与"研究合规"和最小必要原则严重背离。`RedactionEngine.kt:53-56` 直接丢弃 password 节点正是对此的正确防御。

### 6.3 立场

**drop-all-text 不是 idea 的退化，而是对 idea 的纠正**：它在不损失路由/认证判别力的前提下，消除了最大的一类隐私风险。idea-step2 的"采文字"应被**否决并改写**（见 §9）。唯一值得商榷的折中是：是否保留**极少数结构性语义信号**（如 `viewIdResourceName` 已保留、`class_name` 已保留），而这些恰恰已经在采——无需触碰用户内容文本。

## 9. 对 idea 措辞的修正建议

建议把 **step2** 由：

> ~~"端侧用可云端更新的固定策略脱敏，默认采集 UI 组件的文字信息（邮箱、用户名、密码等文字内容）、组件类型、深度等，但不采集前端正文文本"~~

改写为：

> **"端侧丢弃全部可见/输入文本（含正文、输入框、内容描述、窗口标题），仅采集**文本无关的 UI 结构与元数据**：组件类型（`class_name`）、开发者资源 ID（`viewIdResourceName`）、层级深度、子节点数、状态布尔（可点/可编辑/可滚动等）、动作摘要、粗粒度位置（量化网格），以及**仅表示"是否存在文本/内容描述"的布尔标志**（`has_text`/`has_content_description`）；password 节点整节点丢弃。结合运动传感器（加速度计/陀螺仪/磁力计）与触控时序，作为上下文路由与行为认证的输入。云端规则保留为可选的二次防护接口，但默认策略即为端侧 drop-all。"**

理由：(1) 与代码现状（`RedactionEngine.kt`/`JsonCodec.kt` drop-all）一致；(2) 与隐私/合规目标一致（§6）；(3) 不损失路由/认证判别力（文本对二者均非必需）；(4) 消除"采密码明文"这一不可接受的风险。

并建议把 **step5** 的措辞从"提升准确率"明确为**可证伪命题**：在补齐多用户 + 多 session + 冒充协议后，验证"按上下文路由后的专属专家"相对"单一全局模型"在 **EER/FAR@FRR** 等认证指标上的提升，并以无会话泄漏的 GroupKFold 评估。

## 10. 结论速览（一页纸）

1. **整体满足度（一句话）**：**层 A（含"采文字"的字面 idea）不满足且不应满足；层 B（上下文路由提升认证）部分满足——路由原型数据够起步，端到端认证因单用户/无冒充者/触控空而当前不可达。**
2. **step1 满足**（UI 结构采集健康，670 事件/33490 节点/空事件仅 1.49%）。
3. **step2 核心矛盾**：idea 要"采文字"，代码做"drop-all-text"（0/33490 节点含文本）。**这是对 idea 的纠正而非退化**；server `default_rules.json` 仍残留 step2 原意但已完全失效。
4. **step3 部分满足**：LZ4 压缩 + SHA-256 + TLS 到位；负载级"加密"缺位（`encryption="none"`）。
5. **step4 部分满足**：路由**输入端齐备**（包名+结构+短窗传感器），**模型端为零**；现成 `estimated_context_category` 的"100% 命中"是**循环论证**（`ContextFeatureExtractor.kt:31-32` 直抄标签），真·启发式**只覆盖 C2/C3/C4 三类**（C2 占 79.5%，C6 因 `media_like_score≡0` 永不触发）。
6. **step5 最大缺口（不满足）**：**device_id 唯一 = 单用户无冒充者 → 认证任务无法定义**；触控 42 批全空且 API 无坐标；每类仅 ~4–19s 且**只有 1 个 session**。
7. **P0 硬约束**：单用户 + 每类单 session，使得**连无泄漏的路由分类器评估都做不出**（GroupKFold 无法同时满足"不泄漏"与"每折覆盖 8 类"）。
8. **采文本的裁决**：对路由无必要、对认证非主流且高风险；**维持 drop-all-text**。
9. **最小必要改动（P0）**：补多用户多设备、每类多 session、明确认证正负样本与冒充协议——这三者不补，step5 永远不可达。
10. **可证伪锚点**：剔除 `estimated_context_category`/`game_like_score` 两列、在无会话泄漏划分下，若结构+传感器特征仍能把 8 类分到 ≫12.5%，则路由正面成立——**当前数据无法完成此检验**。
